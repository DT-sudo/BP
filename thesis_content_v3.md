4 Практическая часть

4.1 Аргументация выбора технологического стека

В данной главе фиксируются инженерные решения, определившие реализацию прототипа. Стек выбирался не по критерию популярности, а по соответствию задаче: быстро реализовать систему планирования смен с простой эксплуатацией, прозрачной архитектурой и возможностью дальнейшего развития.

4.1.1 Критерии выбора

Выбор технологий определялся требованиями предметной области. Прототип должен был поддерживать аутентификацию, роли, календарное планирование, проверку ограничений при назначении сотрудников и работу с персональными данными, оставаясь при этом простым в сопровождении и развёртывании.

Ключевыми критериями стали: скорость практической реализации, операционная простота, архитектурная выразительность, безопасность и поддерживаемость. Для дипломного проекта было важно не только описать архитектуру, но и довести её до рабочего состояния в ограниченные сроки. Поэтому приоритет отдавался средствам, которые предоставляют готовую инфраструктуру, не требуют сложного окружения и позволяют явно разделить Data, Application и Presentation. Дополнительным аргументом были встроенные механизмы аутентификации, защиты от типовых веб-угроз и низкий порог входа для последующего сопровождения.

4.1.2 Обоснование серверного стека

В качестве серверной платформы был выбран Python с Django. Фреймворк предоставляет маршрутизацию, ORM, шаблоны, формы, аутентификацию, middleware и защиту от типовых угроз, что сокращает объём вспомогательного кода и позволяет сосредоточиться на логике планирования смен.

Для проекта особенно важны три свойства Django. Во-первых, ORM позволяет явно выразить связи между пользователем, должностью, сменой, назначением и недоступностью и закрепить часть инвариантов на уровне моделей. Во-вторых, встроенная аутентификация закрывает критичные задачи ролевого доступа, хранения паролей и работы с сессиями без собственной реализации. В-третьих, формы отделяют первичную обработку входа от более глубоких бизнес-проверок сервисного слоя. В совокупности это делает Django подходящим не только для быстрого старта, но и для наглядной реализации трехслойной архитектуры в одном монолите.

4.1.3 Обоснование клиентского стека и базы данных

Клиентская часть реализована на HTML-шаблонах Django, CSS и Vanilla JavaScript без отдельного фронтенд-фреймворка. Это решение принято сознательно: основная сложность сосредоточена в серверных правилах планирования. Использование React или Vue увеличило бы инфраструктурную сложность без сопоставимой пользы.

Серверный рендеринг сохраняет прозрачный поток данных: сервер формирует HTML, а JavaScript добавляет локальную интерактивность календаря, модальных окон и фильтров. Отказ от сборщика и npm уменьшает число зависимостей и упрощает развёртывание. В качестве базы данных выбрана SQLite, так как для локального прототипа с умеренным объёмом данных она не требует отдельного сервера и сразу интегрируется с Django. При этом использование ORM сохраняет возможность дальнейшего перехода на PostgreSQL без радикальной переработки прикладного кода.

В работе они разделены на технические и нетехнические: первые задают функциональность и инженерные механизмы, вторые — условия эксплуатации, сопровождения и практической применимости прототипа.

4.2.2 Нефункциональные требования

Нетехнические требования задают контекст практического использования и критерии качества, не сводимые только к перечню функций.

NT1. Решение должно быть экономически оправданным: минимальное число внешних зависимостей и возможностью локального запуска без сложной инфраструктуры.

NT2. Интерфейс прототипа должен быть понятен пользователю без длительного обучения.

NT3. Поведение системы должно быть предсказуемым и прозрачным. При ошибке пользователь должен получать понятное сообщение, а не сталкиваться с эффектом «чёрного ящика». 

NT4. Проект должен быть сопровождаемым. Структура исходного кода должна позволять локализовать изменения внутри соответствующего слоя без необходимости переработки всей системы при добавлении новых функций.

NT5. Архитектурные слои системы должны иметь явные границы ответственности и не подменять функции друг друга, включая распределение проверок и валидации по уровням системы в соответствии с их назначением.

4.3 Проектирование архитектуры

После постановки требований была выбрана трёхслойная архитектура: Data, Backend (Application) и Frontend Presentation. Речь идёт о логическом, а не физическом разделении — проект развёртывается как единое Django-приложение, но код организован по принципу separation of concerns с явным разграничением обязанностей между слоями.

4.3.1 Основания выбора трехслойной архитектуры

Выбор трехслойной архитектуры обусловлен предметной областью и структурой требований. Система планирования смен одновременно работает с устойчивыми данными, прикладными правилами и пользовательскими сценариями — каждая из этих областей развивается независимо. Например, правило проверки пересечений относится к application слою, а переработка календаря к presentation слою.

Поэтому разделение на Data, Application и Presentation поддерживает требование T9 о многоуровневой валидации, не даёт критичной логике раствориться во view или JavaScript. Любую новую функцию можно последовательно рассмотреть на уровне данных, операций и интерфейса.

4.3.2 Границы ответственности слоев в текущем проекте

как эти границы работают именно в.
В данном проекте слой Data закрепляет структуру хранения, связи и инварианты, которые должны соблюдаться независимо от того, из какого сценария пришёл запрос. Иными словами, на этом уровне фиксируется, какое состояние система вообще считает допустимым.

Backend (Application) в текущем проекте является центром прикладной координации. Именно здесь пользовательское действие интерпретируется как бизнес-сценарий: создание, изменение, публикация, проверка ограничений, контроль доступа и обработка ошибок. Этот слой решает, может ли конкретное действие быть выполнено корректно в рамках правил системы.

Frontend Presentation в прототипа отвечает за то, чтобы эти сценарии были доступны и понятны пользователю в форме календаря, фильтров, форм и сообщений обратной связи. Его задача — организовать удобное взаимодействие для менеджера и сотрудника, сократить количество лишних действий и сделать состояние системы наглядным. Однако интерфейс не принимает окончательных решений о допустимости операций, а лишь подготавливает и отправляет запросы, оставляя авторитетную проверку backend-слою.

4.3.3 Взаимодействие слоев в проекте

Взаимодействие слоёв в организовано как последовательное прохождение запроса от интерфейса к прикладной логике и далее к данным. HTTP-запрос поступает в Django, маршрутизируется к view, затем application-слой выполняет валидацию и вызывает сервисные или координирующие функции, которые работают с ORM-моделями и базой данных. Результат возвращается обратно в view и далее в HTML или JSON.

Хотя слои физически находятся в одном монолите, точки их взаимодействия остаются чёткими: Presentation не работает с базой напрямую, Data не знает о маршрутах и шаблонах, а Application связывает пользовательское действие с устойчивым состоянием системы. Эта логика определяет и порядок дальнейшего изложения: данные, затем сценарии, затем интерфейс.

4.4 Data

Глава Data описывает формирование модели данных проекта. На этом этапе важно было не только выбрать таблицы и поля, но и закрепить такие связи и ограничения, на которые смогут опираться верхние слои архитектуры.

4.4.1 Проектирование предметных сущностей

Data-слой формировался из ключевых вопросов предметной области: кто работает в системе, какие должности существуют, какие смены создаются, кто на них назначен и в какие дни сотрудник недоступен. Отсюда были выделены сущности `User`, `Position`, `Shift`, `Assignment` и `EmployeeUnavailability`.

Сущность `User` отображает всех пользоваптелей системы. Django рекомендует переопределять модель пользователя в начале проекта, в связи с чем 'User' была имплементрована как расширенный Django класс 'AbstractUser'. Дополненные поля 'role', 'employee_id', 'position' позволяют выразить ролевой доступ и квалификацию без лишних промежуточных таблиц, а свойства `is_manager` 'is_employee' и  инкапсулируют проверку роли.

```python
class User(AbstractUser):
    role = models.CharField(max_length=20, choices=UserRole.choices, default=UserRole.EMPLOYEE)
    employee_id = models.CharField(max_length=20, unique=True, default=generate_employee_id, editable=False)
    position = models.ForeignKey(
        "scheduling.Position",
        on_delete=models.SET_NULL,
        null=True,
        blank=True,
        related_name="employees",
    )
    @property
    def is_manager(self) -> bool:
        return self.role == UserRole.MANAGER
    @property
    def is_employee(self) -> bool:
        return self.role == UserRole.EMPLOYEE
```

`Shift` отображает рабочую смену и содержит поля хранящие дату, интервал, требуемую должность, вместимость, статус жизненного цикла и ссылку на создателя, сохраняя историческую привязку расписания.

```python
class Shift(models.Model):
    date = models.DateField()
    start_time = models.TimeField()
    end_time = models.TimeField()
    position = models.ForeignKey(Position, on_delete=models.PROTECT, related_name="shifts")
    capacity = models.PositiveIntegerField(default=1)
    status = models.CharField(max_length=20, choices=ShiftStatus.choices, default=ShiftStatus.DRAFT)
    created_by = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.PROTECT)
```

'User' и 'Shift' связаны отношением многие-ко-многим, однако вместо скрытой таблицы `ManyToManyField`, которую Django имплементирует автоматически при явной связи многие-ко-многим, была введена отдельная модель `Assignment`. Это сделало назначение самостоятельной записью предметной области: можно явно получать все назначения сотрудника или все назначения на смену, удалять конкретное назначение и при необходимости добавлять к нему собственные атрибуты. Скрытая many-to-many-таблица такой гибкости не даёт, поскольку остаётся лишь внутренней технической связью ORM.

Сущность `Position` реализована как справочник должностей. Поле `name` задаёт уникальное наименование должности, а `is_active` позволяет выводить должность из текущего использования без удаления связанных данных. Такое решение важно для сопровождения: система сохраняет исторические связи сотрудников и смен с уже созданной должностью.

Сущность `EmployeeUnavailability` отображает недоступность сотрудника. Она оформлена как отдельная модель, связывающая конкретного сотрудника с конкретной датой. Это позволило сделать отметки недоступности частью устойчивого состояния системы и проверять их теми же серверными механизмами. 

4.4.2 Структура связей и поведение при удалении

`Position` связана с `User` и `Shift` по схеме один-ко-многим. `Shift` связана с `User` дважды: через `created_by` — с менеджером-создателем, и через `Assignment` с назначенными сотрудниками. `EmployeeUnavailability` связывает `User` с конкретной датой недоступности. Для каждой внешнего ключа в Django задаётся параметр `on_delete` отвечающее за поведение при удалении связанной записи. Определение этого параметра решает, можно ли вообще удалить родительский объект и что происходит с зависимыми данными: `SET_NULL` обнуляет ссылку, сохраняя запись; `PROTECT` запрещает удаление, пока существуют связанные записи; `CASCADE` удаляет зависимые записи вместе с родительской.

Связь сотрудника с должностью описана следующим образом:

```python
position = models.ForeignKey(
    "scheduling.Position",
    on_delete=models.SET_NULL,
    null=True,
    blank=True,
    related_name="employees",
)
```

Здесь выбран `SET_NULL`, поскольку сотрудник должен сохраняться в системе даже в том случае, если его прежняя должность больше не используется. Удаление или деактивация записи справочника не должны приводить к удалению самой учётной записи.

Для смены связь с должностью и создавшим её менеджером задаётся так:

```python
position = models.ForeignKey(
    Position,
    on_delete=models.PROTECT,
    related_name="shifts"
)
created_by = models.ForeignKey(
    settings.AUTH_USER_MODEL,
    on_delete=models.PROTECT,
    related_name="created_shifts",
)
```

В обоих случаях используется `PROTECT`, так как смена не должна остаться без обязательного контекста. Если бы такие записи можно было удалить, в расписании появились бы смены без должности или без указания менеджера, который их создал, что нарушило бы интерпретируемость данных.

Связь назначения сотрудника на смену и связь отметки недоступности с сотрудником описаны иначе:

```python
shift = models.ForeignKey(
    Shift,
    on_delete=models.CASCADE,
    related_name="assignments"
)
employee = models.ForeignKey(
    settings.AUTH_USER_MODEL,
    on_delete=models.CASCADE,
    related_name="assignments"
)
employee = models.ForeignKey(
    settings.AUTH_USER_MODEL,
    on_delete=models.CASCADE,
    related_name="unavailability",
)
```

Здесь используется `CASCADE`, потому что и `Assignment`, и `EmployeeUnavailability` являются зависимыми сущностями. Назначение не имеет смысла без конкретной смены и конкретного сотрудника, а отметка недоступности не существует отдельно от сотрудника. Поэтому при удалении родительской записи такие зависимые записи должны удаляться автоматически.

4.4.3 Ограничения на уровне дата слоя

После определения сущностей были определены инварианты, которые должны сохранять модель в корректном состоянии независимо от пользовательского сценария. Существенная часть таких гарантий реализована непосредственно в моделях.

Ограничения уникальности реализованы через `UniqueConstraint` в `Meta` двух моделей. В `Assignment` это ограничение запрещает назначить одного сотрудника на одну смену более одного раза:

```python
class Assignment(models.Model):
    ...
    class Meta:
        constraints = [
            models.UniqueConstraint(
                fields=["shift", "employee"],
                name="unique_employee_per_shift"
            ),
        ]
```

В `EmployeeUnavailability` аналогичное ограничение запрещает создать две отметки недоступности для одного сотрудника на один день:

```python
class EmployeeUnavailability(models.Model):
```

```python
def _validate_time_range_and_capacity(*, start_time, end_time, capacity) -> None:

    if start_time and end_time and start_time >= end_time:

    if capacity is not None and capacity < 1:

    if errors:

            end_time=self.end_time,
```

Тем самым Data-слой формирует минимальный набор гарантий, на котором строится прикладная логика.

4.5 Backend (Application)

После построения слоя Data следующим этапом стала реализация Application слоя. По мере проектирования стало очевидно, что валидация входных данных, проверка бизнес-правил, координация нескольких операций и обработка HTTP-запросов решают разные задачи и требуют разного уровня абстракции. Поэтому backend-слой был разделён на четыре зоны ответственности: формы (`forms.py`) отвечают за валидацию и очистку входных данных; сервисы (`services.py`) — за бизнес-правила; координирующие функции (`use_cases.py`) — за объединение нескольких операций в один сценарий; представления (`views/`) — за приём запроса и возврат ответа.

4.5.1 Валидация входных данных

Первым шагом разработки application-слоя стала серверная обработка входных данных. На этом этапе HTTP-запрос превращается в проверенные и нормализованные данные, пригодные для передачи в сервисный слой.

Эта задача реализована через классы форм. Формы были введены для моделей взаимодействие с которыми происходи напрямую через интерфейс `User`, `Position` и `Shift`. Сущность `Assignment` управляется косвенно — через поле `employee_ids` формы смены, а `EmployeeUnavailability` передаётся простым запросом с одной датой без отдельной формы.

Для простых сценариев достаточно стандартного `ModelForm`, как в случае с `PositionForm`. Для сотрудников и смен потребовалась дополнительная нормализация ввода. Так, базовая логика обработки данных пользователя была вынесена в `EmployeeBaseForm`, где выполняются приведение email к единому виду, разбор полного имени и единая подготовка данных перед сохранением.

```python
class EmployeeBaseForm(forms.ModelForm):
    full_name = forms.CharField(label="Full name", max_length=150)

    def clean_email(self):
        email = (self.cleaned_data.get("email") or "").strip().lower()
        qs = User.objects.filter(email=email)
        if self.instance and self.instance.pk:
            qs = qs.exclude(pk=self.instance.pk)
        if qs.exists():
            raise ValidationError("An employee with this email already exists.")
        return email
    def save(self, commit=True) -> User:
        user = super().save(commit=False)
        first_name, last_name = _split_full_name(self.cleaned_data.get("full_name"))
        user.first_name = first_name
        user.last_name = last_name
        user.username = self.cleaned_data["email"]
        if commit:
            user.save()
        return user
```

От `EmployeeBaseForm` далее были выделены два класса — `CreateEmployeeForm` и `UpdateEmployeeForm`, поскольку сценарии создания и редактирования сотрудника различаются. Для модели `Shift` была реализована отдельная форма, которая валидирует поля смены и одновременно принимает дополнительные параметры, необходимые для полного сценария планирования.

В классе `ShiftForm` метод `__init__` формирует поле выбора сотрудников, а метод `save()` выставляет смене статус в зависимости от значения `publish`. 

```python
class ShiftForm(forms.ModelForm):
    publish = forms.BooleanField(required=False)
    employee_ids = forms.TypedMultipleChoiceField(coerce=int, required=False)

    class Meta:
        model = Shift
        fields = ["date", "start_time", "end_time", "position", "capacity"]
    def __init__(self, *args, employees=None, **kwargs):
        super().__init__(*args, **kwargs)
        if employees:
            self.fields["employee_ids"].choices = [
                (e.id, e.get_full_name() or e.username) for e in employees
            ]
    def save(self, commit=True):
        instance = super().save(commit=False)
        instance.status = (
            ShiftStatus.PUBLISHED if self.cleaned_data.get("publish") else ShiftStatus.DRAFT
        )
        if commit:
            instance.save()
        return instance
```

Таким образом, формы в проекте решают одну архитектурную задачу: изолируют обработку входных данных от последующей бизнес-логики.

4.5.2 Сервисный слой: бизнес-логика

Следующим шагом стало выделение сервисного слоя в отдельный файл `services.py`. Это позволило собрать ключевые правила планирования в одном месте и отделить их от частных механизмов ввода и HTTP-обработки.

Именно модуль `services.py` реализует жёсткие ограничения предметной области, связанные с назначением сотрудников на смены. Центральной функцией этого модуля является `assign_employees_to_shift`, в которой сосредоточены основные проверки, выполняемые при назначении на смену. Каждое из этих ограничений оформлено отдельной проверкой:

- `_check_position_match` — сотрудник может быть назначен только на смену, соответствующую его должности; назначение «не по профилю» отклоняется;
- `_check_capacity` — количество назначенных не может превышать вместимость смены, заданную менеджером при создании;
- `_check_availability` — система отклоняет назначение, если сотрудник заранее отметил этот день как недоступный;
- `_check_no_overlap` — проверяет отсутствие у сотрудника других смен с пересекающимся временным интервалом в тот же день.

```python
def assign_employees_to_shift(shift: Shift, employee_ids: list[int]) -> None:
    employee_ids = _normalize_employee_ids(employee_ids)
    _check_position_match(shift, employee_ids)
    _check_capacity(shift, employee_ids)
    _check_availability(shift, employee_ids)
    _check_no_overlap(shift, employee_ids)
    _sync_assignments(shift, employee_ids)
```
Финальная функция `_sync_assignments` приводит назначения в базе к выбранному менеджером составу как единый набор — он удаляет снятые назначения и добавляет новые за один проход, исключая промежуточные состояния с частично устаревшим составом смены.(аргументируй лаконично зачем почему это важно)

Проверка пересечения временных интервалов смен реализована на уровне сервисного слоя, что позволяет централизованно применять это ограничение во всех сценариях назначения.(аргументируй лаконично зачем почему это важно)

```python
def _check_no_overlap(shift: Shift, employee_ids: list[int]) -> None:
    conflict = (
        Assignment.objects.filter(
            employee_id__in=employee_ids,
            shift__date=shift.date,
            shift__start_time__lt=shift.end_time,
            shift__end_time__gt=shift.start_time,
        )
        .exclude(shift_id=shift.id)
        .select_related("shift__position")
        .first()
    )
    if not conflict:
        return
    overlapping = conflict.shift
    start = overlapping.start_time.strftime("%H:%M")
    end = overlapping.end_time.strftime("%H:%M")
    raise ValidationError(
        f"Employee already assigned to: {overlapping.position} {start}–{end}"
    )
```

Централизация этих правил в сервисном слое делает поведение системы единообразным во всех пользовательских сценариях. Сервисный слой в проекте также содержит функции выборки данных для календарных представлений — прежде всего `shifts_for_manager()` и `shifts_for_employee()`, которые отбирают смены по периоду, роли пользователя и условиям отображения.

```python
def shifts_for_manager(*, manager_id, start, end, position_ids=None,
                        status=None, understaffed_only=False):
    qs = Shift.objects.filter(
        created_by_id=manager_id, date__gte=start, date__lte=end
    ).select_related("position")
    if position_ids:
        qs = qs.filter(position_id__in=position_ids)
    if status in (ShiftStatus.DRAFT, ShiftStatus.PUBLISHED):
        qs = qs.filter(status=status)
    if understaffed_only:
        qs = qs.annotate(assigned_total=models.Count("assignments")).filter(
            assigned_total__lt=models.F("capacity")
        )
    return qs
```

Парная функция `shifts_for_employee` работает аналогично, но предназначена для сотрудника: она возвращает QuerySet смен за указанный период, жёстко ограниченный статусом `PUBLISHED`. Это означает, что черновые смены физически недоступны сотруднику на уровне запроса к базе данных, а не скрыты в интерфейсе.

4.5.3 Координация сценариев и транзакционные операции

Сервисные функции проверяют отдельные правила, но не выполняют сценарии целиком. Поэтому поверх них был выделен координирующий уровень, функции которого последовательно вызывают несколько сервисных функций и объединяют их результат в одно завершённое действие пользователя.

Наиболее показателен сценарий сохранения смены: он включает валидацию формы, сохранение смены, синхронизацию назначений и возврат единого результата для вызывающего view. Ключевое требование здесь — смена и её назначения должны сохраняться как единая неделимая операция: либо всё записывается успешно, либо при ошибке не сохраняется ничего. В противном случае возможна ситуация, когда смена уже создана, а назначения к ней ещё нет.

Для такого сценария был введён тип результата `SaveShiftResult`. Он нужен как единый способ вернуть итог операции: смена либо сохранена успешно, либо пользователь должен получить сообщение об ошибке. Благодаря этому view не анализирует отдельно форму, сохранение модели и назначение сотрудников, а работает с одним итогом сценария.

```python
@dataclass(frozen=True)
class SaveShiftResult:
    shift: Shift | None
    error: str | None = None

    @property
    def ok(self) -> bool:
        return self.shift is not None and self.error is None
```

Следующий фрагмент показывает ключевую идею `save_shift()`: координирующая функция последовательно объединяет проверку формы, сохранение смены и назначение сотрудников.

```python
def save_shift(*, shift: Shift, post_data) -> SaveShiftResult:
    employees = User.objects.filter(role=UserRole.EMPLOYEE, is_active=True)
    form = ShiftForm(post_data, instance=shift, employees=employees)
    if not form.is_valid():
        return SaveShiftResult(shift=None, error=_first_form_error(form))

    try:
        with transaction.atomic():
            saved_shift = form.save()
            employee_ids = form.cleaned_data.get("employee_ids") or []
            assign_employees_to_shift(saved_shift, employee_ids)
        return SaveShiftResult(shift=saved_shift)
    except Exception as exc:
        return SaveShiftResult(shift=None, error=f"Could not save shift: {exc}")
```

В этом решении важны два аспекта. Во-первых, `transaction.atomic()` защищает от частичного сохранения: в базе либо появляются и сама смена, и её назначения, либо не записывается ничего. Во-вторых, вызов `assign_employees_to_shift()` остаётся изолированной функцией, поэтому правила назначения сосредоточены в одном месте, а координирующая функция лишь собирает их в завершённый пользовательский сценарий.

Отдельной функцией оформлена и массовая публикация смен за период. В этом случае изменяется не одна смена, а сразу группа смен менеджера в выбранном диапазоне дат: все смены со статусом `DRAFT` переводятся в состояние `PUBLISHED`. Это важно как самостоятельный пользовательский сценарий, который не следует смешивать с логикой сохранения одной конкретной смены.

Таким образом, координирующий уровень связывает локальные сервисные проверки с завершёнными действиями пользователя и делает backend-слой ближе к реальным бизнес-сценариям.

4.5.4 Контроль доступа и обработка HTTP-запросов

Завершающим этапом реализации application-слоя стала организация HTTP-обработки и ролевого контроля доступа. Первой задачей было определить, как разграничить доступ между менеджером и сотрудником, не дублируя одинаковые проверки в каждой view-функции. Решением стал параметризованный декоратор `role_required`, из которого выводятся конкретные ограничения для каждой роли.

Благодаря иммплементации ролевой проверки через декоратор обавление новой роли сводится к одной строке без изменения логики проверки. Редирект вместо HTTP 403 выбран намеренно: в двухролевой системе попадание на чужой маршрут — навигационная ситуация, а не ошибка, поэтому пользователь сразу переводится в свой рабочий контекст.

```python
def role_required(user_attr: str, redirect_to: str):
    def decorator(view_func):
        @wraps(view_func)
        def _wrapped(request, *args, **kwargs):
            if not request.user.is_authenticated:
                return redirect("login")
            if not getattr(request.user, user_attr, False):
                return redirect(redirect_to)
            return view_func(request, *args, **kwargs)
        return _wrapped
    return decorator

manager_required = role_required("is_manager", "employee_shifts")
employee_required = role_required("is_employee", "manager_shifts")
```

Ролевой декоратор проверяет тип пользователя, но не принадлежность конкретной смены текущему менеджеру: менеджер мог бы обратиться к смене другого менеджера, зная её идентификатор. Поэтому каждый запрос на получение смены по идентификатору включает условие `created_by=request.user`. С точки зрения безопасности это важно потому, что такая проверка предотвращает нарушение доступа. 

Отдельным решением в области безопасности стал однократный показ сгенерированного пароля: при создании сотрудника пароль записывается в сессию и извлекается только один раз. Открытый пароль нигде не хранится — в базе остаётся только его хэш, что исключает компрометацию паролей даже при утечке данных.

По мере роста числа view-функций их размещение в одном файле стало нецелесообразным. Согласно принципу единственной ответственности (SRP) они были разнесены по модулям: `manager_shifts.py` — операции со сменами, `manager_resources.py` — управление должностями, `employee.py` — расписание сотрудника, `helpers.py` — вспомогательные функции, общие для нескольких модулей.

View-функции минималистичны: они принимают запрос, вызвывают функцию из `services.py` или `use_cases.py` и возвращают ответ. Для всех POST-операций применяется паттерн Post/Redirect/Get — это исключает повторную отправку формы при обновлении страницы.

```python
@manager_required
@require_POST
def save_shift(request: HttpRequest, shift_id: int | None = None) -> HttpResponse:
    is_update = shift_id is not None
    shift = _get_manager_shift_or_404(request, shift_id) if is_update else Shift(created_by=request.user)
    return _save_shift_from_post(
        request,
        shift=shift,
        success_message="Shift updated." if is_update else "Shift created.",
    )
```

Единственным исключением из паттерна PRG стал маршрут отметки недоступности сотрудника. Для этого сценария backend возвращает не редирект, а `JsonResponse`, поскольку результат операции должен быть передан в интерфейс немедленно, без повторной загрузки страницы.

При этом backend по-прежнему выполняет все обязательные проверки: нельзя отметить прошедший день или день с уже назначенной сменой. Тем самым даже в более интерактивном пользовательском сценарии окончательное решение о допустимости действия остаётся на стороне сервера

```python
if day <= today:
    return JsonResponse({"ok": False, "error": "Only dates from tomorrow onwards can be marked as unavailable."}, status=400)

has_shift = Assignment.objects.filter(employee_id=request.user.id, shift__date=day).exists()
if has_shift:
    return JsonResponse({"ok": False, "error": "You have a shift assigned on this day."}, status=400)
```

4.6 Frontend Presentation
 
Финальным этапом разработки стала реализация presentation-слоя. Его задача — отобразить данные и организовать работу с ними так, чтобы пользователь видел состояние расписания, быстро выполнял типовые действия. 

При проектировании интерфейса рассматривались два подхода: отдельное SPA-приложение с API и серверный рендеринг HTML с локальной клиентской интерактивностью. Первый вариант потребовал бы более сложной клиентской архитектуры и дублирования части состояния между сервером и браузером. Для данного проекта это избыточно ввиду того, что число экранов невелико, а основная интерактивность сосредоточена в календаре, фильтрах и модальных окнах.

Поэтому в проекте выбран серверный рендеринг (SSR) на базе Django-шаблонов. Сервер формирует исходную страницу и передаёт авторитетное состояние, а JavaScript добавляет только локальные функции, что позволяет сохранить presentation-слой как самостоятельный логический уровень архитектуры. Интерфейс менеджера при этом разделён на разделы `Shifts` и `Team`, а для сотрудника реализована отдельная страница 'My shifts'. 

4.6.1 Структура шаблонов и статических ресурсов

Структура presentation-слоя проектировалась так, чтобы интерфейс оставался сопровождаемым при сохранении серверного рендеринга. Поэтому HTML, CSS и JavaScript разделены по зонам ответственности. Для менеджера и сотрудника выделены отдельные страницы, а повторяющиеся HTML элементы интерфейса вынесены в директорию `partials`, чтобы переиспользовать их в нескольких шаблонах и изменять в одном месте, что уменьшает дублирование кода, упрощает сопровождение и поддерживает единый внешний вид страниц.

Для страницы `Shifts` отдельно выделены функции работы со временем и конфигурацией, рендеринг календаря, элементы управления и логика модальных окон. Такое разбиение позволяет изменять отдельные части интерфейса без переработки всей страницы.

В рамках Django Templates view подготавливает данные, передаёт их в шаблон, а сервер возвращает пользователю уже сгененрированную HTML-страницу. Для страницы `Shifts` в шаблон передаются режим календаря, опорная дата и сериализованные данные смен и сотрудников. После этого JavaScript не выполняет повторную начальную загрузку, а использует уже переданное сервером состояние для локальной интерактивности календаря. Тем самым frontend остаётся встроенным в серверный маршрут, но сохраняет собственную зону ответственности на уровне представления.

4.6.2 Интерфейс менеджера: раздел Shifts

В разделе `Shifts` сосредоточены календарь, фильтры, операции со сменами и инструменты контроля загруженности персонала.

Сначала были реализованы два календарных режима: недельный и месячный. Недельный вид используется для детального планирования по часам, тогда как месячный даёт обзор периода и позволяет быстро оценить распределение смен по дням.

Недельный режим потребовал отдельного алгоритмического решения, поскольку при плотном расписании пересекающиеся смены накладывались друг на друга. Для устранения этой проблемы была реализована функция `computeShiftLaneLayout`, которая раскладывает временные интервалы по колонкам.

Решение основано на жадном подходе: смены сначала сортируются по времени начала, после чего для каждой из них ищется первая уже освободившаяся колонка. Если такой колонки нет, создаётся новая. В коде это выражено следующим фрагментом:

```javascript
const items = (shifts || [])
    .map((s) => ({
        id: String(s.id),
        start: parseTimeToMinutes(s.start_time),
        end: parseTimeToMinutes(s.end_time),
    }))
    .sort((a, b) => a.start - b.start || a.end - b.end || a.id.localeCompare(b.id));

items.forEach((it) => {
    let laneIndex = -1;
    for (let i = 0; i < laneEnds.length; i++) {
        if (it.start >= laneEnds[i]) {
            laneIndex = i;
            break;
        }
    }
    if (laneIndex === -1) {
        laneIndex = laneEnds.length;
        laneEnds.push(it.end);
    } else {
        laneEnds[laneIndex] = it.end;
    }
    laneById.set(it.id, laneIndex);
});
```

Здесь массив `laneEnds` хранит время окончания последней смены в каждой колонке, поэтому алгоритм работает не со всеми попарными сравнениями, а с текущим состоянием уже занятых колонок. Если несколько смен пересекаются в пределах одного дня, каждая получает отдельную колонку до появления свободной; например, пять одновременных пересечений дадут пять колонок.
Функция `computeShiftLaneLayout` вычисляет схему размещения — номер колонки для каждой смены и общее число колонок за день. Затем эта схема используется при отрисовке недельного столбца: непересекающиеся смены переиспользуют одну колонку, а конфликтующие отображаются раздельно. Благодаря этому календарь остаётся читаемым даже при высокой плотности расписания.

4.6.3 Интерфейс менеджера: раздел Team

Раздел `Team` выделен отдельно, поскольку управление персоналом — самостоятельная задача, отделённая от календарного планирования.

Основой раздела является табличный список сотрудников. В строке сотрудника показываются его атрибуты и меню действий, что позволяет менеджеру быстро найти нужного работника и при необходимости сразу открыть нужный сценарий — просмотр, редактирование, удаление или сброс пароля.

С технической точки зрения здесь использовано решение, отличное от отдельного запроса на каждое открытие карточки сотрудника: необходимые данные для редактирования сохраняются в `data-*` атрибутах строки таблицы и считываются JavaScript при открытии модального окна. Для данного проекта это оправдано, поскольку набор редактируемых полей невелик и уже присутствует на странице после серверного рендеринга. В результате модальное окно открывается без дополнительного обращения к серверу, а логика остаётся простой.

4.6.4 Интерфейс сотрудника

Его задача — предоставить сотруднику возможность видеть свои опубликованные смены и отмечать недоступность, поэтому основой страницы является личный календарь с месячным отображением. По сравнению с интерфейсом менеджера этот экран намеренно проще: сотруднику не требуется управлять структурой расписания, поэтому presentation-логика сведена к просмотру и одной интерактивной операции.

Отметка недоступности реализована через Fetch API: клиент отправляет POST-запрос и после успешного ответа сервера JavaScript обновляет состояние ячейки без перезагрузки страницы. При этом на клиенте хранится только локальное представление выбранных дат, а не самостоятельная бизнес-модель. После подтверждения от сервера интерфейс обновляет множество недоступных дней и повторно отрисовывает календарную сетку на основе уже загруженного состояния. Это позволяет сохранить отзывчивость интерфейса без введения сложного клиентского состояния.

```python
day = date.fromisoformat(request.POST.get("date", ""))
today = date.today()

if day <= today:
    return JsonResponse(
        {"ok": False, "error": "Only dates from tomorrow onwards can be marked."},
        status=400,
    )

has_shift = Assignment.objects.filter(
    employee_id=request.user.id,
    shift__date=day,
).exists()
if has_shift:
    return JsonResponse(
        {"ok": False, "error": "You have a shift assigned on this day."},
        status=400,
    )
```
