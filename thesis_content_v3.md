4 Практическая часть

4.1 Аргументация выбора технологического стека

В данной главе фиксируются инженерные решения, определившие реализацию прототипа. Стек выбирался не по критерию популярности, а по соответствию задаче: быстро реализовать систему планирования смен с простой эксплуатацией, прозрачной архитектурой и возможностью дальнейшего развития.

4.1.1 Критерии выбора

Выбор технологий определялся требованиями предметной области. Прототип должен был поддерживать аутентификацию, роли, календарное планирование, проверку ограничений при назначении сотрудников и работу с персональными данными, оставаясь при этом простым в сопровождении и развёртывании.

Ключевыми критериями стали: скорость практической реализации, операционная простота, архитектурная выразительность, безопасность и поддерживаемость. Для дипломного проекта было важно не только описать архитектуру, но и довести её до рабочего состояния в ограниченные сроки. Поэтому приоритет отдавался средствам, которые предоставляют готовую инфраструктуру, не требуют сложного окружения и позволяют явно разделить Data, Application и Presentation. Дополнительным аргументом были встроенные механизмы аутентификации, защиты от типовых веб-угроз и низкий порог входа для последующего сопровождения.

4.1.2 Обоснование серверного стека

В качестве серверной платформы был выбран Python с Django. Фреймворк предоставляет маршрутизацию, ORM, шаблоны, формы, аутентификацию, middleware и защиту от типовых угроз, что сокращает объём вспомогательного кода и позволяет сосредоточиться на логике планирования смен.

Для проекта особенно важны три свойства Django. Во-первых, ORM позволяет явно выразить связи между пользователем, должностью, сменой, назначением и недоступностью и закрепить часть инвариантов на уровне моделей. Во-вторых, встроенная аутентификация закрывает критичные задачи ролевого доступа, хранения паролей и работы с сессиями без собственной реализации. В-третьих, формы отделяют первичную обработку входа от более глубоких бизнес-проверок сервисного слоя. В совокупности это делает Django подходящим не только для быстрого старта, но и для наглядной реализации трехслойной архитектуры в одном монолите.

4.1.3 Обоснование клиентского стека и базы данных

Клиентская часть реализована на HTML-шаблонах Django, CSS и Vanilla JavaScript без отдельного фронтенд-фреймворка. Это решение принято сознательно: основная сложность сосредоточена в серверных правилах планирования. Использование React или Vue увеличило бы инфраструктурную сложность без сопоставимой пользы.

Серверный рендеринг сохраняет прозрачный поток данных: сервер формирует HTML, а JavaScript добавляет локальную интерактивность календаря, модальных окон и фильтров. Отказ от сборщика и npm уменьшает число зависимостей и упрощает развёртывание. В качестве базы данных выбрана SQLite, так как для локального прототипа с умеренным объёмом данных она не требует отдельного сервера и сразу интегрируется с Django. При этом использование ORM сохраняет возможность дальнейшего перехода на PostgreSQL без радикальной переработки прикладного кода.

4.2 Постановка требований

После выбора технологической базы необходимо зафиксировать требования, относительно которых оценивается корректность проектных решений. В работе они разделены на Функциональные и неФункциональные: первые задают функциональность и инженерные механизмы, вторые — условия эксплуатации, сопровождения и практической применимости прототипа.

4.2.1 Функциональные требования

Функциональные требования определяют, какие функции и какие инженерные свойства должны быть реализованы непосредственно в системе.

T1. Система должна поддерживать обязательную аутентификацию и авторизация пользователей.

Пользователь вводит логин и пароль. Система проверяет учётные данные и перенаправляет его на страницу согласно роли.

T2. Система должна обеспечивать управление профилями сотрудников со стороны менеджера: создание учётных записей сотрудников, назначение должности и генерацию учётных данных для входа.

Менеджер создаёт учётную запись сотрудника, назначает должность. Система генерирует учётные данные и отображает пароль единожды. Менеджер может редактировать профиль или удалить сотрудника.

T3. Система должна поддерживать управление списком должностей.

Менеджер добавляет, переименовывает или удаляет должность в справочнике.

T4. Система должна поддерживать создание и редактирование смен с обязательными атрибутами: дата, время начала, время окончания, должность, вместимость. Новые смены должны создаваться в состоянии черновика.

Менеджер заполняет форму смены: дата, время, должность, вместимость. Смена сохраняется как черновик и не видна сотрудникам.

T5. Система должна поддерживать публикацию смен. Только опубликованные смены должны становиться видимыми сотруднику в личном календаре.

Менеджер публикует одну смену или все черновики текущего периода. Опубликованные смены становятся видимыми в календаре сотрудника.

T6. Система должна предоставлять менеджеру календарный интерфейс планирования и просмотра смен в недельном и месячном представлениях, а сотруднику — персональный календарь опубликованных смен в месячном представлении.

Менеджер просматривает смены в недельном или месячном представлении, фильтрует по должности и статусу. Сотрудник видит только свои опубликованные смены в месячном представлении.

T7. Система должна поддерживать отметку дней недоступности сотрудником. Отметка допустима только для будущих дат и только при отсутствии назначенной смены на выбранный день.

Сотрудник отмечает день как недоступный — только для будущих дат и при отсутствии назначенной смены. Повторный клик снимает отметку. При нарушении условий система отклоняет действие с указанием причины.

T8. Система должна поддерживать назначение сотрудников на смены с проверкой жёстких ограничений: соответствие должности, отсутствие недоступности сотрудника на дату смены, отсутствие временного пересечения с другой сменой и соблюдение лимита вместимости.

Менеджер назначает сотрудника на смену. Система проверяет соответствие должности, отсутствие недоступности, отсутствие пересечений по времени и соблюдение лимита вместимости. При нарушении любого из условий назначение отклоняется с указанием причины.


4.2.2 Нефункциональные требования

Нетехнические требования задают контекст практического использования и критерии качества, не сводимые только к перечню функций.

NT1. Решение должно соответсовать уровню функционально прототипа: минимальное число внешних зависимостей и возможностью локального запуска без сложной инфраструктуры.

NT2. Интерфейс прототипа должен быть понятен пользователю без длительного обучения.

NT3. Поведение системы должно быть предсказуемым и прозрачным. При ошибке пользователь должен получать понятное сообщение.

NT4. Проект должен быть сопровождаемым. Структура исходного кода должна позволять локализовать изменения внутри соответствующего слоя без необходимости переработки всей системы при добавлении новых функций.

NT5. Архитектурные слои системы должны иметь явные границы ответственности и не подменять функции друг друга, включая распределение проверок и валидации по уровням системы в соответствии с их назначением.

4.3 Проектирование архитектуры

После постановки требований и выбора технологического стека была выбрана архитектура для реализации прототипа. Проект развёртывается как единое Django-приложение с явным разграничением обязанностей между слоями.
Изложение материала организовано в соответствии с архитектурными слоями — от данных к логике и далее к интерфейсу — что отражает логическую зависимость слоёв, а не строгую хронологию разработки.

4.3.1 Основания выбора архитектуры

Трёхслойная архитектура была выбрана как наиболее соответствующая задаче: в системе планирования смен требования к хранению данных, бизнес-правилам и интерфейсу меняются по разным причинам и с разной частотой. Например, правило проверки пересечений смен принадлежит application-слою и не должно дублироваться в шаблонах или JavaScript. Переработка календарного представления затрагивает только presentation-слой и не требует изменений в логике валидации.

4.3.2 Границы ответственности и взаимодействие слоев

В реализации функционального прототипа Data слой представлен Django-моделями, закрепляющими структуру таблиц, внешние ключи и ограничения уровня базы данных — в частности, защиту от удаления должности при наличии связанных смен. Валидация на уровне моделей гарантирует целостность независимо от источника запроса.

Application-слой реализован через Django-формы и сервисные функции. Формы выполняют первичную проверку входных данных, модуль `services.py` содержит бизнес-правила: проверку пересечений смен, логику публикации, контроль назначений. View-функции координируют сценарии — получают данные из форм, вызывают сервисы и формируют ответ — но не содержат бизнес-логики напрямую.

Presentation-слой включает Django-шаблоны и клиентский JavaScript. Шаблоны отвечают за формирование HTML-страниц менеджера и сотрудника. JavaScript реализует интерактивность календаря, фильтрацию и модальные окна на клиенте, однако не принимает решений о допустимости операций — окончательная проверка остаётся за backend-слоем.

HTTP-запрос поступает в Django, маршрутизируется к соответствующей view-функции, которая передаёт данные в форму и сервисные функции. Сервисный модуль выполняет бизнес-проверки и работает с ORM-моделями. Результат возвращается в view и далее рендерится в HTML или возвращается как JSON. 

4.4 Datovy layer

Глава описывает формирование модели данных проекта. На этом этапе важно было не только выбрать таблицы и поля, но и закрепить такие связи и ограничения, на которые смогут опираться верхние слои архитектуры.

4.4.1 Проектирование предметных сущностей

Проектирование слоя данных основывалось на анализе предметной области: были выделены ключевые сущности — модель `User` (пользователи системы), модель `Position` (справочник должностей), модель `Shift` (рабочая смена), модель `Assignment` (назначение сотрудника на смену) и модель `EmployeeUnavailability` (отметка недоступности сотрудника).

Модель `User` отображает всех пользователей системы. Django рекомендует переопределять модель пользователя в начале проекта, в связи с чем `User` была реализована как расширение класса `AbstractUser` — базового класса Django для пользователей с поддержкой аутентификации. Дополнительные поля `role`, `employee_id`, `position` позволяют выразить ролевой доступ и квалификацию без лишних промежуточных таблиц, а свойства `is_manager` и `is_employee` инкапсулируют проверку роли.

```python
class User(AbstractUser):
    role = models.CharField(max_length=20, choices=UserRole.choices, default=UserRole.EMPLOYEE)
    employee_id = models.CharField(max_length=20, unique=True, default=generate_employee_id, editable=False)
    position = models.ForeignKey(
        "scheduling.Position",
        on_delete=models.SET_NULL,
        null=True,
        blank=True,
        related_name="employees",
    )
    @property
    def is_manager(self) -> bool:
        return self.role == UserRole.MANAGER
    @property
    def is_employee(self) -> bool:
        return self.role == UserRole.EMPLOYEE
```

Модель `Shift` отображает рабочую смену и содержит поля, хранящие дату, интервал, требуемую должность, вместимость, статус жизненного цикла и ссылку на создателя, сохраняя историческую привязку расписания.

```python
class Shift(models.Model):
    date = models.DateField()
    start_time = models.TimeField()
    end_time = models.TimeField()
    position = models.ForeignKey(Position, on_delete=models.PROTECT, related_name="shifts")
    capacity = models.PositiveIntegerField(default=1)
    status = models.CharField(max_length=20, choices=ShiftStatus.choices, default=ShiftStatus.DRAFT)
    created_by = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.PROTECT)
```

`User` и `Shift` связаны отношением многие-ко-многим, однако вместо поля `ManyToManyField` — стандартного механизма Django для неявной промежуточной таблицы — была введена отдельная модель `Assignment`. Это сделало назначение самостоятельной записью предметной области: можно явно получать все назначения сотрудника или все назначения на смену, удалять конкретное назначение и при необходимости добавлять к нему собственные атрибуты. Неявная many-to-many-таблица такой гибкости не даёт, поскольку остаётся лишь внутренней технической связью ORM.

Модель `Position` реализована как справочник должностей. Поле `name` задаёт уникальное наименование должности, а поле `is_active` позволяет выводить должность из текущего использования без удаления связанных данных. Такое решение важно для сопровождения: система сохраняет исторические связи сотрудников и смен с уже созданной должностью.

Модель `EmployeeUnavailability` отображает недоступность сотрудника. Она оформлена как отдельная модель, связывающая конкретного сотрудника с конкретной датой. Это позволило сделать отметки недоступности частью устойчивого состояния системы и проверять их теми же серверными механизмами. 

4.4.2 Структура связей и поведение при удалении

`Position` связана с `User` и `Shift` по схеме один-ко-многим. `Shift` связана с `User` дважды: через `created_by` — с менеджером-создателем, и через `Assignment` с назначенными сотрудниками. `EmployeeUnavailability` связывает `User` с конкретной датой недоступности. Для каждой внешнего ключа в Django задаётся параметр `on_delete` отвечающее за поведение при удалении связанной записи. Определение этого параметра решает, можно ли вообще удалить родительский объект и что происходит с зависимыми данными: `SET_NULL` обнуляет ссылку, сохраняя запись; `PROTECT` запрещает удаление, пока существуют связанные записи; `CASCADE` удаляет зависимые записи вместе с родительской.

Для связи сотрудника с должностью выбран `SET_NULL`, поскольку сотрудник должен сохраняться в системе даже в том случае, если его прежняя должность больше не используется. Удаление или деактивация записи справочника не должны приводить к удалению самой учётной записи.

```python
position = models.ForeignKey(
    "scheduling.Position",
    on_delete=models.SET_NULL,
    null=True,
    blank=True,
    related_name="employees",
)
```

Для смены связь с должностью и создавшим её менеджером используется `PROTECT`, так как смена не должна остаться без обязательного контекста. Если бы такие записи можно было удалить, в расписании появились бы смены без должности или без указания менеджера, который их создал, что нарушило бы интерпретируемость данных.

```python
position = models.ForeignKey(
    Position,
    on_delete=models.PROTECT,
    related_name="shifts"
)
created_by = models.ForeignKey(
    settings.AUTH_USER_MODEL,
    on_delete=models.PROTECT,
    related_name="created_shifts",
)
```

Связь назначения сотрудника на смену и связь отметки недоступности с сотрудником описаны иначе. Здесь используется `CASCADE`, потому что и `Assignment`, и `EmployeeUnavailability` являются зависимыми сущностями. Назначение не имеет смысла без конкретной смены и конкретного сотрудника, а отметка недоступности не существует отдельно от сотрудника. Поэтому при удалении родительской записи такие зависимые записи должны удаляться автоматически. 

```python
shift = models.ForeignKey(
    Shift,
    on_delete=models.CASCADE,
    related_name="assignments"
)
employee = models.ForeignKey(
    settings.AUTH_USER_MODEL,
    on_delete=models.CASCADE,
    related_name="assignments"
)
employee = models.ForeignKey(
    settings.AUTH_USER_MODEL,
    on_delete=models.CASCADE,
    related_name="unavailability",
)
```

4.4.3 Ограничения на уровне дата слоя

После определения сущностей были определены инварианты, которые должны сохранять модель в корректном состоянии независимо от пользовательского сценария. Существенная часть таких инвариантов реализована непосредственно в моделях.

Ограничения уникальности реализованы через `UniqueConstraint` — декларативный механизм Django для ограничений уникальности на уровне базы данных — объявленный в классе `Meta` двух моделей. В `Assignment` это ограничение запрещает назначить одного сотрудника на одну смену более одного раза:

```python
class Assignment(models.Model):
    ...
    class Meta:
        constraints = [
            models.UniqueConstraint(
                fields=["shift", "employee"],
                name="unique_employee_per_shift"
            ),
        ]
```

В `EmployeeUnavailability` аналогичное ограничение запрещает создать две отметки недоступности для одного сотрудника на один день:

```python
class EmployeeUnavailability(models.Model):
    ...
    class Meta:
        constraints = [
            models.UniqueConstraint(
                fields=["employee", "date"],
                name="unique_employee_unavailability_day"
            ),
        ]
```

Дополнительно на уровне модели `Shift` закреплены базовые проверки интервала времени и вместимости. Эти проверки относятся именно к Data-слою, потому что описывают допустимое состояние сущности независимо от конкретного пользовательского сценария.

```python
def _validate_time_range_and_capacity(*, start_time, end_time, capacity) -> None:
    errors: dict[str, str] = {}
    if start_time and end_time and start_time >= end_time:
        errors["end_time"] = "End time must be after start time."
    if capacity is not None and capacity < 1:
        errors["capacity"] = "Capacity must be at least 1."
    if errors:
        raise ValidationError(errors)

class Shift(models.Model):
    ...
    def clean(self) -> None:
        _validate_time_range_and_capacity(
            start_time=self.start_time,
            end_time=self.end_time,
            capacity=self.capacity,
        )
```

Тем самым Data-слой формирует минимальный набор гарантий, на котором строится прикладная логика.

4.5 Applikacni layer 

Глава описывает реализацию Application-слоя — прикладной логики, которая опирается на модели данных и обслуживает пользовательские сценарии. По мере проектирования стало очевидно, что Applikacni логика неоднородна и требует разного уровня абстракции в зависимости от решаемой задачи. Поэтому Applikacni слой был разделён на четыре зоны ответственности: формы (`forms.py`) отвечают за валидацию и очистку входных данных; сервисы (`services.py`) — за бизнес-правила; координирующие функции (`use_cases.py`) — за объединение нескольких операций в один сценарий; представления (`views/`) — за приём запроса и возврат ответа.

4.5.1 Валидация входных данных

Первым шагом разработки application слоя стала серверная обработка входных данных. На этом этапе HTTP-запрос превращается в проверенные и нормализованные данные, пригодные для передачи в сервисные функции.

Эта задача реализована через классы форм. Формы были введены для моделей, взаимодействие с которыми происходит напрямую через интерфейс: `User`, `Position` и `Shift`. Модель `Assignment` управляется косвенно — через поле `employee_ids` формы смены, а `EmployeeUnavailability` передаётся простым запросом с одной датой без отдельной формы.

Для простых сценариев, где поля формы напрямую соответствуют полям модели, достаточно стандартного Django-класса `ModelForm` — базового класса для форм, автоматически генерирующих поля из модели, — как в случае с `PositionForm`, формой для управления должностями. Для сотрудников и смен потребовалась дополнительная нормализация ввода. Так, базовая логика обработки данных пользователя была вынесена в `EmployeeBaseForm` — общий базовый класс форм сотрудника, — где выполняются приведение email к единому виду, разбор полного имени и единая подготовка данных перед сохранением.

```python
class EmployeeBaseForm(forms.ModelForm):
    full_name = forms.CharField(label="Full name", max_length=150)

    def clean_email(self):
        email = (self.cleaned_data.get("email") or "").strip().lower()
        qs = User.objects.filter(email=email)
        if self.instance and self.instance.pk:
            qs = qs.exclude(pk=self.instance.pk)
        if qs.exists():
            raise ValidationError("An employee with this email already exists.")
        return email
    def save(self, commit=True) -> User:
        user = super().save(commit=False)
        first_name, last_name = _split_full_name(self.cleaned_data.get("full_name"))
        user.first_name = first_name
        user.last_name = last_name
        user.username = self.cleaned_data["email"]
        if commit:
            user.save()
        return user
```

От `EmployeeBaseForm` были выделены два класса. `CreateEmployeeForm` — форма создания нового сотрудника — расширяет базовую форму логикой инициализации: при сохранении новому пользователю присваивается роль `EMPLOYEE` и отзываются административные привилегии. `UpdateEmployeeForm` — форма редактирования существующего сотрудника — наследует базовую форму без изменений, поскольку при редактировании данная инициализация не требуется. 

Для модели `Shift` была реализована форма `ShiftForm` — форма создания и редактирования смены, — которая валидирует поля смены и одновременно принимает дополнительные параметры, необходимые для полного сценария планирования.
В `ShiftForm` метод `__init__` формирует поле выбора сотрудников, а метод `save()` выставляет смене статус в зависимости от значения флага `publish`. 

```python
class ShiftForm(forms.ModelForm):
    publish = forms.BooleanField(required=False)
    employee_ids = forms.TypedMultipleChoiceField(coerce=int, required=False)

    class Meta:
        model = Shift
        fields = ["date", "start_time", "end_time", "position", "capacity"]
    def __init__(self, *args, employees=None, **kwargs):
        super().__init__(*args, **kwargs)
        if employees:
            self.fields["employee_ids"].choices = [
                (e.id, e.get_full_name() or e.username) for e in employees
            ]
    def save(self, commit=True):
        instance = super().save(commit=False)
        instance.status = (
            ShiftStatus.PUBLISHED if self.cleaned_data.get("publish") else ShiftStatus.DRAFT
        )
        if commit:
            instance.save()
        return instance
```

4.5.2 Сервисный модуль: бизнес-логика

Бизнес-правила предметной области сосредоточены в модуле `services.py`, отделённом от логики обработки запросов и представлений.

Именно модуль `services.py` реализует жёсткие ограничения предметной области, связанные с назначением сотрудников на смены. Центральной функцией этого модуля является `assign_employees_to_shift`, в которой сосредоточены основные проверки, выполняемые при назначении на смену. Каждое из этих ограничений оформлено отдельной проверкой:

`_check_position_match` — сотрудник может быть назначен только на смену, соответствующую его должности; назначение «не по профилю» отклоняется;
`_check_capacity` — количество назначенных не может превышать вместимость смены, заданную менеджером при создании;
`_check_availability` — система отклоняет назначение, если сотрудник заранее отметил этот день как недоступный;
`_check_no_overlap` — проверяет отсутствие у сотрудника других смен с пересекающимся временным интервалом в тот же день.

```python
def assign_employees_to_shift(shift: Shift, employee_ids: list[int]) -> None:
    employee_ids = _normalize_employee_ids(employee_ids)
    _check_position_match(shift, employee_ids)
    _check_capacity(shift, employee_ids)
    _check_availability(shift, employee_ids)
    _check_no_overlap(shift, employee_ids)
    _sync_assignments(shift, employee_ids)
```
Финальная функция `_sync_assignments` приводит назначения в базе к выбранному менеджером составу как единый набор — он удаляет снятые назначения и добавляет новые за один проход, исключая промежуточные состояния с частично устаревшим составом смены. 

Проверка пересечения временных интервалов смен реализована на уровне сервисного слоя, что позволяет централизованно применять это ограничение во всех сценариях назначения. 

```python
def _check_no_overlap(shift: Shift, employee_ids: list[int]) -> None:
    conflict = (
        Assignment.objects.filter(
            employee_id__in=employee_ids,
            shift__date=shift.date,
            shift__start_time__lt=shift.end_time,
            shift__end_time__gt=shift.start_time,
        )
        .exclude(shift_id=shift.id)
        .select_related("shift__position")
        .first()
    )
    if not conflict:
        return
    overlapping = conflict.shift
    start = overlapping.start_time.strftime("%H:%M")
    end = overlapping.end_time.strftime("%H:%M")
    raise ValidationError(
        f"Employee already assigned to: {overlapping.position} {start}–{end}"
    )
```

Модуль `services.py` также содержит функции выборки данных для календарных представлений — прежде всего `shifts_for_manager()` и `shifts_for_employee()`, которые формируют выборку смен с учётом периода, роли пользователя и условий отображения.

```python
def shifts_for_manager(*, manager_id, start, end, position_ids=None,
                        status=None, understaffed_only=False):
    qs = Shift.objects.filter(
        created_by_id=manager_id, date__gte=start, date__lte=end
    ).select_related("position")
    if position_ids:
        qs = qs.filter(position_id__in=position_ids)
    if status in (ShiftStatus.DRAFT, ShiftStatus.PUBLISHED):
        qs = qs.filter(status=status)
    if understaffed_only:
        qs = qs.annotate(assigned_total=models.Count("assignments")).filter(
            assigned_total__lt=models.F("capacity")
        )
    return qs
```

Функция `shifts_for_employee` реализована по аналогичному принципу, однако ориентирована на сотрудника: результирующий `QuerySet` ограничен исключительно сменами со статусом `PUBLISHED`. Тем самым черновые смены исключаются на уровне запроса к базе данных, что обеспечивает их недоступность вне зависимости от логики представления.

4.5.3 Координация сценариев и транзакционные операции

Сервисные функции проверяют отдельные правила, но не выполняют сценарии целиком. Поэтому поверх них был выделен координирующий уровень, функции которого последовательно вызывают несколько сервисных функций и объединяют их результат в одно завершённое действие пользователя.

Наиболее показателен сценарий сохранения смены: он включает валидацию формы, сохранение смены, синхронизацию назначений и возврат единого результата для вызывающего view. Ключевое требование здесь — смена и её назначения должны сохраняться как единая неделимая операция: либо всё записывается успешно, либо при ошибке не сохраняется ничего. В противном случае возможна ситуация, когда смена уже создана, а назначения к ней ещё нет.

Для такого сценария был введён тип результата `SaveShiftResult`. Он нужен как единый способ вернуть итог операции: смена либо сохранена успешно, либо пользователь должен получить сообщение об ошибке. Благодаря этому view не анализирует отдельно форму, сохранение модели и назначение сотрудников, а работает с одним итогом сценария.

```python
@dataclass(frozen=True)
class SaveShiftResult:
    shift: Shift | None
    error: str | None = None

    @property
    def ok(self) -> bool:
        return self.shift is not None and self.error is None
```

Следующий фрагмент кода показывает ключевую идею `save_shift()`: координирующая функция последовательно объединяет проверку формы, сохранение смены и назначение сотрудников.

```python
def save_shift(*, shift: Shift, post_data) -> SaveShiftResult:
    employees = User.objects.filter(role=UserRole.EMPLOYEE, is_active=True)
    form = ShiftForm(post_data, instance=shift, employees=employees)
    if not form.is_valid():
        return SaveShiftResult(shift=None, error=_first_form_error(form))

    try:
        with transaction.atomic():
            saved_shift = form.save()
            employee_ids = form.cleaned_data.get("employee_ids") or []
            assign_employees_to_shift(saved_shift, employee_ids)
        return SaveShiftResult(shift=saved_shift)
    except Exception as exc:
        return SaveShiftResult(shift=None, error=f"Could not save shift: {exc}")
```

Данное решение основано на двух принципах. Применение `transaction.atomic()` обеспечивает атомарность операции: смена и её назначения сохраняются как единое целое, что исключает частичную запись в базу данных. Функция `assign_employees_to_shift()` при этом остаётся самостоятельным модулем с сосредоточенными в нём правилами назначения; координирующая функция лишь объединяет их в рамках полного пользовательского сценария.

Отдельной функцией оформлена и массовая публикация смен за период. В этом случае изменяется не одна смена, а сразу группа смен менеджера в выбранном диапазоне дат: все смены со статусом `DRAFT` переводятся в состояние `PUBLISHED`. Это важно как самостоятельный пользовательский сценарий, который не следует смешивать с логикой сохранения одной конкретной смены.

4.5.4 Контроль доступа и обработка HTTP-запросов

Завершающим этапом реализации application-слоя стала организация HTTP-обработки и ролевого контроля доступа. Первой задачей было определить, как разграничить доступ между менеджером и сотрудником, не дублируя одинаковые проверки в каждой view-функции. Решением стал параметризованный декоратор `role_required`, из которого выводятся конкретные ограничения для каждой роли.

Благодаря иммплементации ролевой проверки через декоратор обавление новой роли сводится к одной строке без изменения логики проверки. Редирект вместо HTTP 403 выбран намеренно: в двухролевой системе попадание на чужой маршрут — навигационная ситуация, а не ошибка, поэтому пользователь сразу переводится в свой рабочий контекст.

```python
def role_required(user_attr: str, redirect_to: str):
    def decorator(view_func):
        @wraps(view_func)
        def _wrapped(request, *args, **kwargs):
            if not request.user.is_authenticated:
                return redirect("login")
            if not getattr(request.user, user_attr, False):
                return redirect(redirect_to)
            return view_func(request, *args, **kwargs)
        return _wrapped
    return decorator

manager_required = role_required("is_manager", "employee_shifts")
employee_required = role_required("is_employee", "manager_shifts")
```

Ролевой декоратор проверяет тип пользователя, но не принадлежность конкретной смены текущему менеджеру: менеджер мог бы обратиться к смене другого менеджера, зная её идентификатор. Поэтому каждый запрос на получение смены по идентификатору включает условие `created_by=request.user`. С точки зрения безопасности это важно потому, что такая проверка предотвращает нарушение доступа. 

Отдельным решением в области безопасности стал однократный показ сгенерированного пароля: при создании сотрудника пароль записывается в сессию и извлекается только один раз. Открытый пароль нигде не хранится — в базе остаётся только его хэш, что исключает компрометацию паролей даже при утечке данных.

По мере роста числа view-функций их размещение в одном файле стало нецелесообразным. Согласно принципу единственной ответственности (SRP) они были разнесены по модулям: `manager_shifts.py` содержит операции со сменами, `manager_resources.py` — управление должностями, `employee.py` — расписание сотрудника, `helpers.py` — вспомогательные функции, общие для нескольких модулей.

View-функции минималистичны: они принимают запрос, вызывают функцию из `services.py` или `use_cases.py` — модуля координирующих функций для составных сценариев — и возвращают ответ. Для всех POST-операций применяется паттерн Post/Redirect/Get — это исключает повторную отправку формы при обновлении страницы.

```python
@manager_required
@require_POST
def save_shift(request: HttpRequest, shift_id: int | None = None) -> HttpResponse:
    is_update = shift_id is not None
    shift = _get_manager_shift_or_404(request, shift_id) if is_update else Shift(created_by=request.user)
    return _save_shift_from_post(
        request,
        shift=shift,
        success_message="Shift updated." if is_update else "Shift created.",
    )
```

Единственным исключением из паттерна PRG стал маршрут отметки недоступности сотрудника. Для этого сценария backend возвращает не редирект, а `JsonResponse` — встроенный класс Django для ответа в формате JSON, — поскольку результат операции должен быть передан в интерфейс немедленно, без повторной загрузки страницы.

При этом backend по-прежнему выполняет все обязательные проверки: нельзя отметить прошедший день или день с уже назначенной сменой. Тем самым даже в пользовательском сценарии окончательное решение о допустимости действия остаётся на стороне сервера

```python
if day <= today:
    return JsonResponse({"ok": False, "error": "Only dates from tomorrow onwards can be marked as unavailable."}, status=400)

has_shift = Assignment.objects.filter(employee_id=request.user.id, shift__date=day).exists()
if has_shift:
    return JsonResponse({"ok": False, "error": "You have a shift assigned on this day."}, status=400)
```

4.6 Presentation layer
 
Глава описывает реализацию presentation-слоя: организацию шаблонов и статических ресурсов, подход к передаче данных через SSR, реализацию интерфейсов менеджера и сотрудника, а также распределение ответственности между сервером и клиентом. 

4.6.1 Структура шаблонов и статических ресурсов

Presentation-слой построен на принципе разделения ответственности. HTML, CSS и JavaScript изолированы друг от друга; повторяющиеся элементы интерфейса вынесены в отдельные фрагменты, что исключает дублирование разметки. Для менеджера и сотрудника предусмотрены отдельные страницы в соответствии с различием их функциональных ролей.

Клиентские JavaScript модули структурированы аналогично: утилиты общего назначения (работа с датами, логика календаря) отделены от страничной логики. Страница `Shifts` выделена в отдельные модули по функциональным областям (рендеринг, элементы управления, модальные окна) ввиду существенно большего объёма клиентской логики по сравнению с остальными страницами. 

4.6.2 SSR

При проектировании интерфейса рассматривались два подхода: SPA и SSR. Первый вариант потребовал бы более сложной клиентской архитектуры и дублирования части состояния между сервером и браузером. Для данного проекта это избыточно ввиду того, что число экранов невелико, а основная интерактивность сосредоточена в календаре, фильтрах и модальных окнах. Поэтому в проекте для основных страниц выбран серверный рендеринг SSR на базе Django-шаблонов. 

Технически это реализовано двумя каналами передачи начального состояния. Первый канал — JSON-блоки в HTML, которые сервер встраивает в шаблон при рендеринге:

```html
<script type="application/json" id="managerShiftsData">{{ shifts_json|safe }}</script>
<script type="application/json" id="managerEmployeesData">{{ employees_json|safe }}</script>
```

Второй канал — `data-*` атрибуты контейнера страницы, где передаются параметры навигации и URL-шаблоны для операций над сменами:

```html
<div
    id="managerShiftPage"
    data-view="{{ view }}"
    data-anchor="{{ anchor|date:'Y-m-d' }}"
    data-shift-update-url-template="{% url 'update_shift' 0 %}"
></div>
```

Функция `parseJsonScript` принимает идентификатор элемента и значение по умолчанию. В случае отсутствия элемента или некорректного JSON возвращается значение по умолчанию, что обеспечивает инициализацию страницы в пустом состоянии без прерывания работы интерфейса:

```javascript
function parseJsonScript(id, fallback) {
    const element = document.getElementById(id);
    if (!element) return fallback;
    try {
        return JSON.parse(element.textContent || '');
    } catch {
        return fallback;
    }
}
```

Описанный подход применён на всех основных страницах приложения, что позволило исключить дополнительные запросы к серверу при первичной загрузке и упростить клиентскую архитектуру.

4.6.3 Клиентская интерактивность и границы ответственности

В текущем проекте интерактивность реализована как слой поверх SSR-страницы, а не как отдельная клиентская бизнес-логика. Это означает, что JavaScript управляет отображением, локальным состоянием и пользовательскими действиями, но не принимает окончательных решений о допустимости операций.

Распределение ответственности в presentation-слое выглядит следующим образом:

- клиент читает уже переданное сервером начальное состояние (JSON-блоки и `data-*` атрибуты);
- клиент управляет локальным UX-состоянием: открытие модальных окон, переключение режимов календаря, подсветка смен, локальная фильтрация и временные представления данных;
- клиент отправляет действия пользователя на backend и интерпретирует ответ как «успех/ошибка» без самостоятельного изменения бизнес-правил;
- backend остаётся единственным авторитетным уровнем валидации ограничений.

Именно поэтому даже в сценариях с мгновенной реакцией интерфейса (например, отметка недоступности через Fetch API) окончательное состояние подтверждается только ответом сервера. Такой подход поддерживает прозрачность архитектуры.

Практически это даёт два эффекта:
Снижается риск расхождения между клиентской и серверной логикой при развитии проекта: правила меняются в одном месте — в backend.
Упрощается сопровождение frontend-кода, так как большинство изменений интерфейса локализуется в шаблонах, стилях и JS-модулях без необходимости пересобирать отдельный SPA-контур.

4.6.4 Интерфейс менеджера: раздел Shifts

Сначала были реализованы два календарных режима: недельный и месячный. Недельный вид используется для детального планирования по часам, тогда как месячный даёт обзор периода и позволяет быстро оценить распределение смен по дням.

Для размещения иконок смен в календарной сетке в недельном режиме потребовал отдельного алгоритмического решения, поскольку при плотном расписании пересекающиеся смены накладывались друг на друга. Для устранения этой проблемы была реализована функция `computeShiftLaneLayout`, которая распределяет смены по колонкам внутри каждого дня. 
Для решения задачи применяется жадный алгоритм (greedy algorithm): смены сортируются по времени начала, затем каждой назначается первая свободная колонка — та, в которой предыдущая смена уже завершилась. Если свободных колонок нет, создаётся новая. Массив `laneEnds` хранит время окончания последней смены в каждой колонке, что позволяет не сравнивать все пары смен, а проверять только текущее состояние занятых колонок. Результат функции — номер колонки для каждой смены и общее число колонок за день; при отрисовке непересекающиеся смены занимают одну колонку, а конфликтующие отображаются раздельно.

```javascript
const items = (shifts || [])
    .map((s) => ({
        id: String(s.id),
        start: parseTimeToMinutes(s.start_time),
        end: parseTimeToMinutes(s.end_time),
    }))
    .sort((a, b) => a.start - b.start || a.end - b.end || a.id.localeCompare(b.id));

items.forEach((it) => {
    let laneIndex = -1;
    for (let i = 0; i < laneEnds.length; i++) {
        if (it.start >= laneEnds[i]) {
            laneIndex = i;
            break;
        }
    }
    if (laneIndex === -1) {
        laneIndex = laneEnds.length;
        laneEnds.push(it.end);
    } else {
        laneEnds[laneIndex] = it.end;
    }
    laneById.set(it.id, laneIndex);
});
```

Функция `applyTimedShiftChipVertical` позиционирует плитку смены в сетке абсолютными координатами: вертикальная позиция и высота вычисляются из времени начала и длительности смены, горизонтальное положение и ширина определяются номером колонки и их общим количеством. Это позволяет визуально отразить и временной диапазон смены, и её место среди пересекающихся смен дня:

```javascript
}
```

С технической точки зрения при открытии формы редактирования сотрудника принято решение не делать отдельный запрос к серверу за данными. Вместо этого необходимые поля для предзаполнения формы встраиваются в data-* атрибуты строки таблицы при серверном рендеринге и считываются JavaScript в момент открытия модального окна. 1. Cервер встраивает в строку таблицы только те поля, которые требуются для предзаполнения формы редактирования:


2. Клиент считывает эти значения непосредственно в обработчике действия и переносит их в поля формы без промежуточного API-вызова:

```javascript

emailInput.value = row.dataset.email || '';
```

3. Клиент параметризует action формы по выбранному сотруднику через серверный URL-шаблон, не дублируя маршруты в JavaScript-коде:


4.6.6 Интерфейс сотрудника

Его задача — предоставить сотруднику возможность видеть свои опубликованные смены и отмечать недоступность, поэтому основой страницы является личный календарь с месячным отображением.

Отметка недоступности реализована через Fetch API: клиент отправляет POST-запрос и после успешного ответа сервера JavaScript обновляет состояние ячейки без перезагрузки страницы. После подтверждения от сервера интерфейс обновляет множество недоступных дней и повторно отрисовывает календарную сетку на основе уже загруженного состояния, что позволяет сохранить отзывчивость интерфейса без введения сложного клиентского состояния.

С технической стороны отправка запроса выполняется через общий helper `postFormJson`, который автоматически добавляет CSRF-токен в заголовок `X-CSRFToken`. После успешного ответа серверный результат синхронизируется с локальным множеством дат `unavailableDays`, затем переотрисовываются список выбранных дат и календарная сетка:

```javascript
const data = await postFormJson(toggleUrl, { date: isoDate });
if (data.unavailable) unavailableDays.add(isoDate);
else unavailableDays.delete(isoDate);
renderUnavailableList();
renderMonthGrid(config, getShiftsData());
```

Это решение демонстрирует выбранный компромисс: клиент обеспечивает отзывчивость интерфейса, но источником истины о допустимости операции остаётся сервер.

В интерактивном сценарии отметки недоступности отправка выполняется единым методом. Код ниже демонстрирует, что добавление CSRF-заголовка и форматирование тела запроса централизованы в одном месте, благодаря чему поведение POST-операций в клиентском слое остаётся единообразным.

```javascript
window.postFormJson = async function (url, data) {
    const csrfToken = window.getCsrfToken?.() || '';
    const response = await fetch(url, {
        method: 'POST',
        headers: {
            'Content-Type': 'application/x-www-form-urlencoded;charset=UTF-8',
            'Accept': 'application/json',
            ...(csrfToken ? { 'X-CSRFToken': csrfToken } : {}),
        },
        body: new URLSearchParams(data || {}),
    });
    return response.json();
};
```

Маршрут операции при этом передаётся сервером через `data-*` атрибут. Такой подход исключает дублирование URL в JavaScript и поддерживает принцип единого источника конфигурации на стороне backend.

```html
<div id="employeeShiftPage"
         data-toggle-url="{{ toggle_url }}"
         data-today="{{ today|date:'Y-m-d' }}"></div>
```
