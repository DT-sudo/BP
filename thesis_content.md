## Functional Requirements (Функциональные требования)

F1 – Авторизация 
В приложении должна быть реализована возможность входа в систему для пользователей, имеющих учётную запись. Для входа используются логин и пароль. Неавторизованным пользователям доступ к страницам расписания или управления не предоставляется. 

Сценарий использования: Пользователь открывает приложение и видит страницу входа. При вводе корректных логина и пароля он перенаправляется на главную страницу в соответствии со своей ролью. При вводе неверных данных отображается сообщение об ошибке, доступ не предоставляется. Прямой переход по URL защищённой страницы без активной сессии перенаправляет пользователя на страницу входа.

F2 – Роли в системе
Приложение должно поддерживать две пользовательские роли: manager & employee, которые определяют какой функционал доступен пользователю.  Менеджеру предоставляется доступ ко всем функциям управления командой, должностями и сменами; сотруднику — ограниченный доступ, позволяющий только просматривать свои смены и отмечать дни недоступности. Совмещение ролей одним пользователем не допускается.

Сценарий использования: Работник входит в систему и видит только свой личный календарь смен. Попытка перейти по адресу страницы управления сменами перенаправляет его обратно на личный календарь. Менеджер, войдя под своей учётной записью, получает доступ к полному интерфейсу: созданию смен, управлению командой и публикации расписания.

F10 – Интерактивный календарь руководителя 
Менеджеру должен быть предоставлен интерфейс в виде сеточной структуры в недельном и месячном представлении. При наложении смен применяется жадный алгоритм размещения в параллельных колонках для исключения визуального перекрытия. Календарём отображаются статус каждой смены и нехватка персонала; поддерживается фильтрация по должностям, заполненности смены и ее статусу draft/published.
Сценарий использования: Менеджер открывает недельный вид на текущую неделю и видит, что в среду утром две смены пересекаются по времени — они автоматически отображаются в соседних колонках без перекрытия. Смена, в которой назначен только один сотрудник при вместимости три, выделена как недоукомплектованная. Применив фильтр по должности «Бариста», менеджер видит только релевантные смены и быстро принимает решение о дополнительном назначении.

F11 – Календарь сотрудника 
Сотрудникам должен быть предоставлен доступ к персональному календарю для просмотра опубликованных менеджером смен.

Сценарий использования: Сотрудник входит в систему и открывает раздел личного календаря. Он видит только опубликованные смены, на которые назначен, и может просматривать их в недельном и месячном представлении. Изменения статуса смен (публикация/снятие с публикации) отражаются в календаре после обновления страницы.

F12 – Отметка о недоступности сотрудника Сотрудникам должна быть предоставлена возможность отмечать в личном календаре дни недоступности. Заявленная недоступность автоматически учитывается системой как жёсткое ограничение при формировании графиков Менеджером.

Сценарий использования: Сотрудник заранее отмечает в календаре 20 и 21 апреля как недоступные дни (например, по причине экзамена). При попытке менеджера назначить сотрудника на смену в эти даты система блокирует назначение и отображает причину конфликта. Если отметка недоступности удаляется, сотрудник снова становится доступен для назначения.

F3 – Регистрация сотрудников
Добавление новых сотрудников должно осуществляться исключительно Менеджером. При создании учётной записи указываются контактные данные и назначается должность; Должна быть предусмотрена автоматическая генерирация  уникального идентификатора сотрудника и одноразовый временный пароль.

Сценарий использования: Менеджер кофейни принимает на работу нового бариста Томаша Крала. В разделе управления командой он заполняет форму: имя, фамилия, e-mail, телефон, должность «Бариста». После сохранения система отображает сгенерированный идентификатор EMP-483921 и временный пароль — единственный раз, после чего они исчезают из интерфейса. Менеджер передаёт эти данные сотруднику лично, и тот может войти в систему.

F6 – Управление должностями 
(hard constrains) для управдения должностями – менеджер не может удалить должность пока в системе существуют работники которые на нее подписаны – допиши а так же оцени на сколько это праивльное решение.
Менеджеру должна быть предоставлена возможность создавать, редактировать и переводить должности в неактивный статус. Наличие должностей выступает базовым критерием соответствия квалификации при валидации назначений сотрудников на смены. Полное удаление должности блокируется системой в случае, если к ней привязаны сотрудники или смены.
Сценарий использования: Менеджер создаёт должность «Кассир». Спустя месяц должность упраздняется — менеджер переводит её в неактивный статус: она перестаёт быть доступной при создании новых смен, однако исторические данные сохраняются. При попытке полностью удалить должность «Повар», к которой привязаны действующие сотрудники, система отображает предупреждение и блокирует удаление.

F7 – Создание смен
Менеджеру должна быть предоставлена возможность создания рабочей смены. Для ее создания необходимо указать следующие обязательные атрибуты: даты, времени начала и окончания, должности и необходимой вместимости. Без указания хотя бы одного обязательного атрибута, создание смены не возможно.  Новые смены сохраняются в статусе «Черновик» (Draft), что позволяет Менеджеру предварительно моделировать график до его публикации сотрудникам, снижая риск организационной путаницы.
Сценарий использования: Менеджер составляет расписание на следующую неделю: создаёт смену 14.04, с 08:00 до 16:00, должность «Бариста», вместимость 2 человека. При попытке указать время окончания раньше времени начала система отображает ошибку валидации. Корректно созданная смена появляется в календаре менеджера со статусом «Черновик».

F8 – Реализация жёстких ограничений
При создании смены менеджером выбирается должность и дата, после чего в поле выбора сотрудников отображаются только те, кто имеет нужную должность и не отметил этот день как недоступный. Дополнительно, независимо от способа отправки данных, серверной стороной выполняется проверка следующих жёстких ограничений (hard constraints):
•	сотрудник не назначен на другую пересекающуюся по времени смену;
•	лимит вместимости конкретной смены не превышен.
В случае конфликта назначение блокируется; причина нарушения отображается пользователю в явном виде.
Сценарий использования: Менеджер создаёт смену на 14 апреля с профилем «Бариста». В списке выбора отображаются только бариста, не отметившие этот день как недоступный. Иван Новак в списке присутствует, менеджер его выбирает. Однако сервер обнаруживает, что у Ивана уже есть пересекающаяся смена с 12:00 до 20:00, и блокирует назначение с указанием причины конфликта.
F9 – Публикация смен 
Менеджеру должна быть предоставлена возможность индивидуально либо массово переводить смены из статуса «Черновик» в статус «Опубликована». Только опубликованные смены отображаются в личных календарях сотрудников.
Сценарий использования: Менеджер завершил составление расписания на неделю и нажимает кнопку «Опубликовать всё». Все черновики текущей недели получают статус «Опубликована» и мгновенно становятся видимы в личных календарях назначенных сотрудников. Одну конкретную смену, требующую уточнения, менеджер оставляет в статусе черновика и публикует отдельно после согласования.

N1 – Экономичность внедрения

Для предотвращения операционных ошибок соблюдение бизнес-правил должно гарантироваться на каждом уровне архитектуры: на уровне клиентского интерфейса, на уровне серверной бизнес-логики и на уровне базы данных. Такой подход обеспечивает целостность и согласованность данных расписания независимо от условий выполнения запросов.

## 4.3 Настройка среды разработки

Прежде чем приступить к реализации функциональности приложения, необходимо было подготовить рабочее окружение и определить базовую структуру проекта. Этот этап закладывает фундамент для всей последующей разработки — от организации файлов до способа взаимодействия между компонентами системы.

### 4.3.1 Инструменты и установка зависимостей

Разработка ShiftFlow велась в среде, типичной для современных Django-проектов. В качестве интерпретатора использовался Python 3.11, установленный в изолированном виртуальном окружении. Изоляция зависимостей через виртуальное окружение — стандартная практика, которая предотвращает конфликты между библиотеками разных проектов и обеспечивает воспроизводимость среды на любой машине.

Инициализация проекта началась с создания Django-проекта командой `django-admin startproject shiftflow`. Эта команда генерирует базовую структуру с конфигурационным модулем, однако стандартная структура была модифицирована для лучшего разделения ответственности, о чём подробнее в следующем подразделе.

Основной зависимостью проекта является Django — единственная внешняя библиотека, необходимая для работы серверной части. В соответствии с выбранным в предыдущей главе технологическим стеком, клиентская часть реализована на чистом JavaScript без дополнительных фреймворков, что исключает необходимость в npm, webpack или других инструментах сборки фронтенда. Такой минималистичный подход к зависимостям упрощает развёртывание и снижает порог входа для новых разработчиков.

Конфигурация проекта сосредоточена в файле `settings.py`, где определены ключевые параметры: подключённые приложения, настройки базы данных, пути к шаблонам и статическим файлам. Важным архитектурным решением стало использование переменных окружения для конфиденциальных настроек:

```python
SECRET_KEY = os.environ.get("SECRET_KEY", "dev-secret-key-change-me")
DEBUG = os.environ.get("DEBUG", "1") == "1"
```

Это позволяет хранить секретные ключи и параметры подключения к базе данных вне репозитория, что соответствует принципам безопасной разработки. Значения по умолчанию обеспечивают работоспособность в режиме локальной разработки без дополнительной конфигурации.

### 4.3.2 Структура проекта и организация директорий

Архитектура файловой системы проекта отражает принцип чёткого разделения серверной и клиентской частей. Вместо стандартной плоской структуры Django-проекта было принято решение вынести backend и frontend в отдельные директории верхнего уровня:

```
BP_proj/
├── backend/
│   ├── manage.py
│   ├── shiftflow/          # конфигурационный модуль
│   │   ├── settings.py
│   │   ├── urls.py
│   │   └── wsgi.py
│   └── apps/               # пакет приложений
│       ├── accounts/       # управление пользователями
│       └── scheduling/     # планирование смен
└── frontend/
    ├── static/
    │   ├── css/
    │   └── js/
    └── templates/
        ├── auth/
        ├── employee/
        ├── manager/
        └── partials/
```

Такое разделение было выбрано по нескольким причинам. Во-первых, оно делает границу между серверным и клиентским кодом явной на уровне файловой системы, что упрощает навигацию по проекту. Во-вторых, подобная структура облегчает потенциальный переход к раздельному развёртыванию — например, при масштабировании через CDN для статических файлов.

Директория `backend/apps/` объединяет все Django-приложения в единый Python-пакет. Это решение отличается от типичного подхода, когда приложения размещаются в корне проекта. Группировка приложений в пакет обеспечивает несколько преимуществ: чистоту корневой директории, явную принадлежность модулей к бизнес-логике проекта, а также единообразные импорты вида `apps.accounts.models` вместо разрозненных `accounts.models`.

Проект разделён на два приложения с чётко разграниченными зонами ответственности:

- **accounts** — управление пользователями, аутентификация, авторизация и ролевая модель. Это приложение отвечает за вопрос «кто использует систему и что ему разрешено».

- **scheduling** — основная бизнес-логика планирования: смены, назначения сотрудников, позиции, управление недоступностью. Это приложение отвечает за вопрос «что система делает».

Такое разделение следует принципу единственной ответственности на уровне модулей. Каждое приложение можно разрабатывать и тестировать относительно независимо, а изменения в логике аутентификации не затрагивают код планирования смен.

Структура фронтенда организована по ролям пользователей: директории `templates/manager/`, `templates/employee/` и `templates/auth/` соответствуют разделам интерфейса для менеджера, сотрудника и страницам аутентификации. Директория `templates/partials/` содержит переиспользуемые компоненты — шапку, иконки, модальные окна. Аналогичная логика применена к JavaScript-файлам в `static/js/`: каждый файл отвечает за интерактивность конкретного раздела интерфейса.

Связь между backend и frontend осуществляется через настройки Django. В `settings.py` указаны пути к шаблонам и статическим файлам:

```python
TEMPLATES = [
    {
        "DIRS": [PROJECT_ROOT / "frontend" / "templates"],
        ...
    }
]
STATICFILES_DIRS = [PROJECT_ROOT / "frontend" / "static"]
```

Использование `PROJECT_ROOT` вместо `BASE_DIR` позволяет корректно разрешать пути к frontend-директории, которая находится на уровень выше backend. Это техническое решение напрямую следует из выбранной файловой структуры.

Подводя итог, настройка среды разработки ShiftFlow была направлена на создание понятной и масштабируемой структуры. Разделение на backend и frontend, группировка приложений в пакет `apps`, организация шаблонов по ролям — все эти решения упрощают сопровождение кода и закладывают основу для дальнейшего развития системы.

## 4.4 Архитектура приложения

Определив структуру проекта, следующим шагом стала разработка архитектуры самого приложения. Архитектурные решения на этом этапе определяют, как компоненты системы взаимодействуют между собой, как организован поток данных и где размещается бизнес-логика.

### 4.4.1 Модульность Django-проекта

Django-проект ShiftFlow состоит из конфигурационного модуля и двух функциональных приложений. Такая модульная структура была выбрана вместо монолитного подхода, при котором весь код размещается в одном приложении.

**Конфигурационный модуль shiftflow** выполняет роль точки входа и координационного центра. Он содержит настройки проекта (`settings.py`), корневую маршрутизацию (`urls.py`) и WSGI-конфигурацию для развёртывания. Этот модуль не содержит бизнес-логики — его задача исключительно техническая: связать компоненты воедино и определить глобальные параметры.

Корневой `urls.py` делегирует обработку запросов приложениям:

```python
urlpatterns = [
    path("admin/", admin.site.urls),
    path("", include("apps.accounts.urls")),
    path("", include("apps.scheduling.urls")),
]
```

Оба приложения подключаются к корню без префикса, что позволяет каждому из них определять собственные URL-паттерны. Такой подход обеспечивает гибкость: при необходимости можно добавить префикс (например, `/api/`) без изменения внутренней маршрутизации приложений.

**Приложение accounts** отвечает за всё, что связано с пользователями: модель пользователя, аутентификацию, авторизацию и управление сотрудниками. Помимо стандартных компонентов Django-приложения (модели, представления, формы), оно содержит два дополнительных модуля:

- `decorators.py` — декораторы для ролевого контроля доступа
- `context_processors.py` — функции, добавляющие данные пользователя в контекст всех шаблонов

Выделение этих модулей в отдельные файлы отражает принцип разделения ответственности: декораторы — это инфраструктурный код, применяемый к представлениям, а контекстные процессоры — механизм интеграции с системой шаблонов.

**Приложение scheduling** реализует основную бизнес-логику планирования смен. Его структура сложнее, чем у accounts, что отражает объём функциональности:

```
scheduling/
├── models.py         # модели данных: Shift, Assignment, Position, ...
├── services.py       # сервисный слой бизнес-логики
├── forms.py          # формы валидации
├── urls.py           # маршрутизация
└── views/            # пакет представлений
    ├── manager_shifts.py
    ├── manager_resources.py
    └── employee.py
```

Представления вынесены в отдельный пакет `views/` вместо единого файла `views.py`. Это решение было принято, когда объём кода представлений превысил 300 строк и стало сложно ориентироваться в файле. Разделение по файлам соответствует логическим группам функциональности: управление сменами, управление ресурсами (позициями), интерфейс сотрудника.

Такая модульная структура обеспечивает несколько преимуществ. Во-первых, каждый модуль можно понять и модифицировать независимо от других. Во-вторых, при расширении системы новая функциональность добавляется в соответствующий модуль или как новое приложение, не затрагивая существующий код. В-третьих, модульность упрощает тестирование: каждый компонент можно тестировать изолированно.

### 4.4.2 Поток запроса и MVT-архитектура

Django реализует паттерн MVT (Model-View-Template), который определяет чёткое разделение ответственности между компонентами. В ShiftFlow этот паттерн был расширен добавлением сервисного слоя, что создаёт следующий поток обработки запроса:

```
HTTP-запрос
    ↓
Middleware (безопасность, сессии, аутентификация)
    ↓
URL Router → выбор View
    ↓
View (обработка запроса, координация)
    ↓
Service (бизнес-логика и валидация)
    ↓
Model (доступ к данным)
    ↓
Template (формирование HTML)
    ↓
HTTP-ответ
```

**Middleware** — первый слой обработки. Django применяет цепочку middleware к каждому запросу. В ShiftFlow используется стандартный набор: `SecurityMiddleware` добавляет заголовки безопасности, `SessionMiddleware` управляет сессиями, `AuthenticationMiddleware` привязывает текущего пользователя к запросу, `CsrfViewMiddleware` защищает от CSRF-атак. Порядок middleware важен: аутентификация должна происходить после восстановления сессии.

**URL Router** направляет запрос к соответствующему представлению на основе URL-паттерна. Маршрутизация в ShiftFlow организована иерархически: корневой `urls.py` делегирует приложениям, которые определяют собственные паттерны. Например, запрос к `/manager/shifts/` проходит через корневой роутер к `apps.scheduling.urls`, где сопоставляется с представлением `manager_shifts`.

**View (представление)** — точка обработки конкретного HTTP-запроса. В ShiftFlow представления реализованы как функции, а не классы. Этот выбор обусловлен характером приложения: большинство представлений выполняют простые операции CRUD, где функциональный подход обеспечивает более явный и читаемый код.

Типичное представление выполняет несколько задач: проверяет права доступа через декораторы, извлекает параметры из запроса, вызывает сервисный слой для бизнес-операций, формирует ответ. Пример структуры представления:

```python
@manager_required
@require_http_methods(["POST"])
def create_shift(request: HttpRequest) -> HttpResponse:
    # Извлечение данных из запроса
    # Вызов сервисного слоя
    # Формирование ответа (HTML или JSON)
```

Декоратор `@manager_required` проверяет, что пользователь аутентифицирован и имеет роль менеджера. Декоратор `@require_http_methods` ограничивает допустимые HTTP-методы. Такой декларативный подход выносит инфраструктурную логику из тела функции.

**Service (сервисный слой)** — архитектурное дополнение к стандартному MVT. Бизнес-логика вынесена в отдельный модуль `services.py` вместо размещения в представлениях или моделях. Это решение имеет несколько обоснований:

- Представления остаются тонкими и отвечают только за HTTP-взаимодействие
- Бизнес-правила собраны в одном месте, что упрощает их понимание и модификацию
- Сервисные функции можно переиспользовать из разных представлений
- При добавлении REST API бизнес-логика не дублируется

Сервисный слой содержит функции валидации и операции над данными:

```python
def validate_employee_no_overlap(employee_id: int, shift: Shift) -> None:
    overlapping = Shift.objects.filter(
        assignments__employee_id=employee_id, 
        date=shift.date
    ).exclude(id=shift.id)
    for other in overlapping:
        if _overlaps(shift.start_time, shift.end_time, other.start_time, other.end_time):
            raise ValidationError(f"Employee already assigned to: {other.position}...")
```

Валидация генерирует исключение `ValidationError`, которое представление перехватывает и преобразует в сообщение пользователю. Такое разделение позволяет сервисному слою оставаться независимым от способа представления ошибки.

**Model (модель)** обеспечивает доступ к данным и инкапсулирует структуру базы данных. Django ORM позволяет работать с данными как с Python-объектами, абстрагируясь от SQL. Модели ShiftFlow содержат не только определение полей, но и методы доступа и бизнес-свойства. Подробно модель данных рассматривается в разделе 4.5.1.

**Template (шаблон)** отвечает за формирование HTML-ответа. Шаблоны Django используют простой язык разметки с переменными, условиями и циклами. Представление передаёт контекст — словарь данных, который шаблон использует для рендеринга. Контекстные процессоры добавляют общие данные (например, информацию о текущем пользователе) во все шаблоны автоматически:

```python
def user_ui_context(request) -> dict[str, Any]:
    user = request.user
    display_name = user.get_full_name() or user.username
    initials = "".join([p[0] for p in display_name.split()[:2] if p])
    return {
        "user_display_name": display_name,
        "user_initials": initials.upper(),
        "user_header_role": "Manager" if user.is_manager else "Employee",
    }
```

Этот контекстный процессор делает данные пользователя доступными в любом шаблоне без явной передачи из каждого представления.

Описанная архитектура обеспечивает чёткое разделение ответственности. Каждый слой решает свою задачу: middleware — инфраструктурную обработку, представления — координацию HTTP-запросов, сервисы — бизнес-логику, модели — доступ к данным, шаблоны — визуальное представление. Такое разделение упрощает понимание, тестирование и модификацию каждого компонента независимо от других.

## 4.5 Серверная часть

Серверная часть ShiftFlow реализует хранение данных, бизнес-логику и обработку HTTP-запросов. Этот раздел рассматривает три ключевых аспекта: структуру данных, логику валидации и операций, а также механизмы обработки запросов и контроля доступа.

### 4.5.1 Модель данных

Модель данных определяет структуру информации в системе и связи между сущностями. В ShiftFlow модели распределены между двумя приложениями в соответствии с их доменной принадлежностью.

**Приложение accounts** содержит единственную модель — `User`. Вместо использования стандартной модели пользователя Django было принято решение создать кастомную модель, наследующую от `AbstractUser`:

```python
class User(AbstractUser):
    role = models.CharField(max_length=20, choices=UserRole.choices, default=UserRole.EMPLOYEE)
    employee_id = models.CharField(max_length=20, unique=True, default=generate_employee_id)
    phone = models.CharField(max_length=50, blank=True)
    position = models.ForeignKey("scheduling.Position", on_delete=models.SET_NULL, null=True)
```

Это решение обусловлено необходимостью расширить стандартную модель пользователя полями, специфичными для предметной области: ролью, идентификатором сотрудника, телефоном и должностью. Создание кастомной модели на этапе инициализации проекта — рекомендуемая практика Django, поскольку изменение модели пользователя после создания миграций значительно сложнее.

Поле `role` использует паттерн `TextChoices` — перечисление, определённое на уровне Python:

```python
class UserRole(models.TextChoices):
    MANAGER = "manager", "Manager"
    EMPLOYEE = "employee", "Employee"
```

Такой подход обеспечивает типобезопасность при работе с ролями в коде и автоматическую валидацию на уровне базы данных. В отличие от хранения ролей в отдельной таблице, `TextChoices` упрощает код и избегает лишних JOIN-запросов, что оправдано при фиксированном наборе ролей.

Модель дополнена вычисляемыми свойствами для удобства проверки роли:

```python
@property
def is_manager(self) -> bool:
    return self.role == UserRole.MANAGER
```

Эти свойства инкапсулируют логику проверки роли и делают код представлений и декораторов более читаемым: `if user.is_manager` вместо `if user.role == UserRole.MANAGER`.

**Приложение scheduling** содержит четыре модели, образующие ядро предметной области.

`Position` — справочник должностей. Модель минималистична: только название и флаг активности. Флаг `is_active` позволяет скрывать должности из интерфейса без удаления, сохраняя ссылочную целостность существующих данных.

`Shift` — центральная сущность системы, представляющая рабочую смену:

```python
class Shift(models.Model):
    date = models.DateField()
    start_time = models.TimeField()
    end_time = models.TimeField()
    position = models.ForeignKey(Position, on_delete=models.PROTECT)
    capacity = models.PositiveIntegerField(default=1)
    status = models.CharField(choices=ShiftStatus.choices, default=ShiftStatus.DRAFT)
    created_by = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.PROTECT)
```

Несколько архитектурных решений требуют пояснения. Поле `capacity` определяет максимальное количество сотрудников на смене. Это позволяет создавать групповые смены (например, когда нужны три бариста одновременно) без дублирования записей.

Поле `status` использует `TextChoices` для статусов «черновик» и «опубликовано». Черновики видны только менеджеру и не отображаются сотрудникам. Такой workflow позволяет планировать расписание заранее и публиковать его, когда оно готово.

Связь с позицией использует `on_delete=models.PROTECT` — при попытке удалить позицию, связанную со сменами, операция будет отклонена на уровне базы данных с формированием исключения. Это предотвращает нарушение целостности данных: если для позиции существуют смены, её нельзя удалить, только деактивировать.

`Assignment` — связующая модель между сменой и сотрудником. Реализует отношение «многие-ко-многим» с дополнительными метаданными:

```python
class Assignment(models.Model):
    shift = models.ForeignKey(Shift, on_delete=models.CASCADE, related_name="assignments")
    employee = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.CASCADE)

    class Meta:
        constraints = [
            models.UniqueConstraint(fields=["shift", "employee"], name="unique_employee_per_shift")
        ]
```

Уникальное ограничение на уровне базы данных гарантирует, что один сотрудник не может быть назначен на одну смену дважды. Это последний рубеж защиты — даже если валидация в коде пропустит дубликат, база данных его отклонит.

`EmployeeUnavailability` хранит даты, когда сотрудник недоступен для работы:

```python
class EmployeeUnavailability(models.Model):
    employee = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.CASCADE)
    date = models.DateField(db_index=True)

    class Meta:
        constraints = [
            models.UniqueConstraint(fields=["employee", "date"], name="unique_employee_unavailability_day")
        ]
```

Индекс по полю `date` ускоряет запросы при проверке доступности сотрудников на конкретную дату — частая операция при назначении на смены.

Связи между моделями образуют следующую структуру:

```
User (accounts)
  │
  ├──→ Position (через position, FK)
  │         │
  │         └──→ Shift (через shifts, reverse FK)
  │                   │
  │                   └──→ Assignment (через assignments, reverse FK)
  │                              │
  └──────────────────────────────┘ (через employee, FK)
  │
  └──→ EmployeeUnavailability (через unavailability, reverse FK)
```

Эта схема позволяет эффективно выполнять типичные запросы: получить все смены сотрудника, всех сотрудников на смене, проверить доступность сотрудника на дату.

### 4.5.2 Сервисный слой и бизнес-логика

Обоснование выделения сервисного слоя было дано в разделе 4.4.2. Здесь рассматривается конкретная реализация бизнес-правил и операций в модуле `services.py`.

Центральная задача сервисного слоя — валидация бизнес-правил при назначении сотрудников на смены. Система проверяет четыре условия.

**Проверка вместимости смены** гарантирует, что количество назначенных сотрудников не превышает capacity:

```python
def validate_shift_capacity(shift: Shift, desired_assigned_count: int) -> None:
    if desired_assigned_count > shift.capacity:
        raise ValidationError("Cannot assign more employees than shift capacity.")
```

**Проверка пересечения смен** предотвращает назначение сотрудника на перекрывающиеся по времени смены:

```python
def validate_employee_no_overlap(employee_id: int, shift: Shift) -> None:
    overlapping = Shift.objects.filter(
        assignments__employee_id=employee_id, 
        date=shift.date
    ).exclude(id=shift.id)
    
    for other in overlapping:
        if _overlaps(shift.start_time, shift.end_time, other.start_time, other.end_time):
            raise ValidationError(f"Employee already assigned to: {other.position}...")
```

Функция `_overlaps` реализует алгоритм проверки пересечения временных интервалов: два интервала пересекаются, если начало первого раньше конца второго И конец первого позже начала второго.

**Проверка доступности сотрудника** использует записи из `EmployeeUnavailability`:

```python
def validate_employee_available(employee_id: int, shift: Shift) -> None:
    if EmployeeUnavailability.objects.filter(employee_id=employee_id, date=shift.date).exists():
        raise ValidationError(f"Employee is unavailable on {shift.date.isoformat()}.")
```

**Проверка соответствия позиции** гарантирует, что сотрудник может работать на смене только своей должности:

```python
def validate_employees_match_shift_position(shift: Shift, employee_ids: list[int]) -> None:
    valid_employee_ids = set(
        User.objects.filter(
            id__in=employee_ids,
            role="employee",
            is_active=True,
            position_id=shift.position_id,
        ).values_list("id", flat=True)
    )
    invalid = [eid for eid in employee_ids if eid not in valid_employee_ids]
    if invalid:
        raise ValidationError("Selected employees must match the shift position.")
```

Все функции валидации генерируют `ValidationError` при нарушении правил. Это стандартное исключение Django, которое представления перехватывают и преобразуют в сообщения для пользователя.

**Атомарные операции** обеспечивают целостность данных при сложных изменениях. Назначение сотрудников на смену объединено в транзакцию:

```python
@transaction.atomic
def set_shift_assignments(shift: Shift, employee_ids: list[int]) -> None:
    employee_ids = list(dict.fromkeys(employee_ids))  # удаление дубликатов
    
    validate_employees_match_shift_position(shift, employee_ids)
    validate_shift_capacity(shift, len(employee_ids))
    
    for employee_id in employee_ids:
        validate_employee_available(employee_id, shift)
        validate_employee_no_overlap(employee_id, shift)

    # Удаляем старые назначения, которых нет в новом списке
    Assignment.objects.filter(shift=shift).exclude(employee_id__in=employee_ids).delete()
    
    # Создаём новые назначения
    existing = set(Assignment.objects.filter(shift=shift).values_list("employee_id", flat=True))
    to_create = [Assignment(shift=shift, employee_id=eid) for eid in employee_ids if eid not in existing]
    Assignment.objects.bulk_create(to_create)
```

Декоратор `@transaction.atomic` гарантирует: если любая валидация сгенерирует исключение, все изменения в базе данных будут отменены. Без транзакции частичное обновление назначений могло бы привести к некорректному состоянию.

**Массовые операции** оптимизируют работу с группами смен. Публикация и удаление черновиков реализованы как batch-операции:

```python
def publish_drafts_in_range(*, manager_id: int, start: date, end: date) -> tuple[list[int], list[int]]:
    qs = _manager_shifts_qs(manager_id=manager_id, start=start, end=end).filter(status=ShiftStatus.DRAFT)
    
    # Блокируем публикацию смен с недоступными сотрудниками
    blocked_ids = shift_ids_blocked_by_unavailability(qs)
    publish_qs = qs.exclude(id__in=blocked_ids)
    
    published_ids = list(publish_qs.values_list("id", flat=True))
    if published_ids:
        publish_qs.update(status=ShiftStatus.PUBLISHED)
    
    return published_ids, blocked_ids
```

Функция возвращает два списка: успешно опубликованные смены и заблокированные. Это позволяет представлению сформировать информативное сообщение: «Опубликовано 5 смен. 2 смены заблокированы из-за недоступных сотрудников».

Метод `update()` выполняет изменение на уровне SQL без загрузки объектов в память, что эффективно при работе с большим количеством записей.

### 4.5.3 Обработка запросов и управление доступом

Этот подраздел рассматривает механизмы приёма HTTP-запросов, формирования ответов и контроля доступа к функциональности.

**Организация URL** следует RESTful-подобной структуре, хотя приложение не является полноценным REST API. URL-паттерны группируются по ролям и ресурсам:

```python

path("manager/employees/<int:user_id>/update/", views.employee_update),
path("manager/employees/<int:user_id>/delete/", views.employee_delete),

# scheduling/urls.py — управление сменами
path("manager/shifts/", views.manager_shifts),
path("manager/shifts/<int:shift_id>/update/", views.update_shift),
path("manager/shifts/<int:shift_id>/delete/", views.delete_shift),
```

Префикс `manager/` явно указывает на принадлежность к интерфейсу менеджера, что упрощает понимание структуры и настройку прав доступа. Идентификаторы ресурсов включены в URL (`<int:shift_id>`), что соответствует принципам адресуемости ресурсов.

**Функциональные представления** обрабатывают запросы и формируют ответы. Выбор функций вместо классов обоснован в разделе 4.4.2. Рассмотрим типичную структуру представления:

```python
@manager_required

    today = timezone.localdate()
    view = (request.GET.get("view") or "week").lower()
    anchor = _parse_date(request.GET.get("date"), today)
    
    if request.method == "POST":
        action = request.POST.get("action")
        if action == "publish":
            return _handle_publish_all(request, start, end)
        # ... другие action
    
    # GET: рендеринг страницы
    shifts_payload = [...]  # подготовка данных
    return render(request, "manager/manager-shifts.html", context)
```

Представление обрабатывает и GET, и POST в одной функции. GET возвращает HTML-страницу, POST выполняет действия (публикация, удаление) и перенаправляет обратно. Такой подход соответствует паттерну Post/Redirect/Get, предотвращающему повторную отправку форм при обновлении страницы.

**Гибридные ответы** — особенность ShiftFlow. Некоторые представления возвращают HTML для полной загрузки страницы и JSON для AJAX-запросов:

```python
@manager_required
@require_http_methods(["GET"])
def shift_details(request: HttpRequest, shift_id: int) -> JsonResponse:
    shift = get_object_or_404(Shift.objects, pk=shift_id, created_by=request.user)
    return JsonResponse({
        "id": shift.id,
        "date": shift.date.isoformat(),
        "start_time": shift.start_time.strftime("%H:%M"),
        # ...
    })
```

JSON-endpoint используется JavaScript-кодом для получения данных смены при открытии модального окна редактирования. Это избавляет от перезагрузки страницы при каждом действии.

**Ролевая авторизация** реализована через декораторы. Базовый декоратор `role_required` параметризован:

```python
def role_required(user_attr: str, redirect_to: str):
    def decorator(view_func):
        @wraps(view_func)
        def _wrapped(request: HttpRequest, *args, **kwargs):
            if not request.user.is_authenticated:
                return redirect("login")
            if not getattr(request.user, user_attr, False):
                return redirect(redirect_to)
            return view_func(request, *args, **kwargs)
        return _wrapped
    return decorator

def manager_required(view_func):
    return role_required("is_manager", "employee_shifts")(view_func)
```

Декоратор `@manager_required` проверяет два условия: аутентификацию и роль. Неаутентифицированный пользователь перенаправляется на страницу входа, аутентифицированный с неподходящей ролью — на страницу своего интерфейса. Такой подход лучше, чем отображение ошибки 403: пользователь сразу попадает туда, где может работать.

**Проверка владельца ресурса** выполняется на уровне представлений. Менеджер может редактировать только смены, которые он создал:

```python
shift = get_object_or_404(Shift.objects, pk=shift_id, created_by=request.user)
```

Функция `get_object_or_404` возвращает 404, если смена не найдена или принадлежит другому менеджеру. Это предотвращает несанкционированный доступ: даже зная ID чужой смены, менеджер не сможет её изменить.

**Формы и серверная валидация** обеспечивают проверку вводимых данных. Django-формы инкапсулируют логику валидации:

```python
class EmployeeBaseForm(forms.ModelForm):
    full_name = forms.CharField(label="Full name", max_length=150)

    def clean_email(self):
        email = (self.cleaned_data.get("email") or "").strip().lower()
        qs = User.objects.filter(email=email)
        if self.instance and self.instance.pk:
            qs = qs.exclude(pk=self.instance.pk)
        if qs.exists():
            raise ValidationError("An employee with this email already exists.")
        return email
```

Метод `clean_email` нормализует email (приведение к нижнему регистру, удаление пробелов) и проверяет уникальность. При редактировании существующего сотрудника исключается его собственная запись из проверки — иначе изменение любого поля вызывало бы ошибку «email уже существует».

Формы отделяют валидацию от представлений. Представление проверяет `form.is_valid()` и либо сохраняет данные, либо возвращает форму с ошибками:

```python
if request.method == "POST":
    form = CreateEmployeeForm(request.POST)
    if form.is_valid():
        employee = form.save()
        messages.success(request, "Employee created.")
        return redirect("manager_employees")
    messages.error(request, "Please fix the errors and try again.")
```

Фреймворк сообщений Django (`messages`) используется для уведомлений. Сообщения сохраняются в сессии и отображаются при следующем рендеринге страницы, что работает корректно даже при перенаправлениях.

Описанная архитектура серверной части обеспечивает надёжную обработку данных, соблюдение бизнес-правил и контроль доступа. Чёткое разделение на модели, сервисы и представления упрощает сопровождение и расширение системы.

## 4.6 Клиентская часть

Клиентская часть ShiftFlow реализует интерактивный интерфейс пользователя. В соответствии с выбранным технологическим стеком, интерфейс построен на Django-шаблонах с дополнением клиентским JavaScript без использования фреймворков. Этот раздел рассматривает архитектуру рендеринга, реализацию календарных представлений, интерактивные компоненты и стилизацию.

### 4.6.1 Гибридная модель рендеринга

ShiftFlow использует гибридную модель, сочетающую серверный рендеринг (SSR) и клиентскую интерактивность. Сервер формирует полную HTML-страницу, которую JavaScript обогащает динамическим поведением без полной перезагрузки.

**Распределение ответственности** между сервером и браузером было определено следующим образом:

Сервер отвечает за:
- Формирование структуры страницы и базовой разметки
- Подготовку данных в виде JSON, встроенного в страницу
- Обработку форм и выполнение бизнес-операций
- Аутентификацию и авторизацию

Браузер отвечает за:
- Рендеринг календарной сетки на основе полученных данных
- Интерактивные элементы: модальные окна, фильтры, выбор сотрудников
- Мгновенную обратную связь без запросов к серверу
- Валидацию на стороне клиента (дублирующую серверную)

Такое разделение выбрано по нескольким причинам. Серверный рендеринг обеспечивает быструю начальную загрузку и корректную работу без JavaScript. Клиентский JavaScript добавляет интерактивность там, где она критична для пользовательского опыта — в календарном интерфейсе и формах создания смен.

**Передача данных** от сервера к JavaScript осуществляется через JSON, встроенный в HTML-страницу:

```html
<script id="managerShiftsData" type="application/json">
  {{ shifts_json|safe }}
</script>
```

JavaScript извлекает данные через вспомогательную функцию:

```javascript
const shifts = parseJsonScript('managerShiftsData', []);
```

Этот подход избегает дополнительного AJAX-запроса при загрузке страницы. Данные доступны сразу, и JavaScript может начать рендеринг без задержки. При этом сохраняется возможность обновления данных через AJAX — например, при открытии модального окна редактирования смены.

**Модульная организация JavaScript** отражает принцип разделения ответственности. Код интерфейса менеджера разбит на модули:

### 4.6.2 Календарные представления

Календарь — центральный элемент интерфейса менеджера. Реализованы три представления: Day (день), Week (неделя) и Month (месяц). Каждое представление оптимизировано для своего сценария использования.

**Week view** — основное рабочее представление, отображающее временную сетку с часовыми интервалами. Алгоритм построения:

1. Создание сетки 7×24 (дни × часы) через CSS Grid
2. Группировка смен по датам
3. Вычисление позиций смен с учётом пересечений (lane layout)
4. Рендеринг chips (карточек смен) с абсолютным позиционированием

Ключевая сложность — размещение пересекающихся по времени смен. Если на один день назначены смены 9:00–12:00 и 10:00–14:00, они должны отображаться рядом, не перекрывая друг друга. Для решения этой задачи реализован алгоритм lane layout:

```javascript
function computeShiftLaneLayout(shifts) {
  const items = shifts
    .map(s => ({
      id: String(s.id),
      start: parseTimeToMinutes(s.start_time),
      end: parseTimeToMinutes(s.end_time),
    }))
    .sort((a, b) => a.start - b.start || a.end - b.end);

  const laneEnds = [];  // время окончания последней смены в каждой полосе
  const laneById = new Map();

  items.forEach(it => {
    // Ищем полосу, где смена поместится без пересечения
    let laneIndex = -1;
    for (let i = 0; i < laneEnds.length; i++) {
      if (it.start >= laneEnds[i]) {
        laneIndex = i;
        break;
      }
    }
    // Если не нашли — создаём новую полосу
    if (laneIndex === -1) {
      laneIndex = laneEnds.length;
      laneEnds.push(it.end);
    } else {
      laneEnds[laneIndex] = it.end;
    }
    laneById.set(it.id, laneIndex);
  });

  return { laneById, laneCount: Math.max(1, laneEnds.length) };
}
```

Алгоритм использует жадную стратегию: для каждой смены ищется первая полоса, где она поместится без пересечения. Если такой полосы нет — создаётся новая. Результат — минимальное количество полос при гарантированном отсутствии пересечений.

Позиционирование смен вычисляется на основе времени и номера полосы:

```javascript
function applyTimedShiftChipVertical(chip, shift, laneIndex, laneCount, hourHeightPx) {
  const start = parseTimeToMinutes(shift.start_time);
  const durationMinutes = parseTimeToMinutes(shift.end_time) - start;

  chip.style.top = `${(start / 60) * hourHeightPx}px`;
  chip.style.height = `${(durationMinutes / 60) * hourHeightPx}px`;

  const pct = 100 / laneCount;
  chip.style.left = `calc(${laneIndex * pct}%)`;
  chip.style.width = `calc(${pct}%)`;
}
```

**Month view** отображает ячейки дней с упрощёнными карточками смен. Здесь нет временной сетки — важна общая картина распределения смен по дням. Смены выводятся списком внутри каждой ячейки.

**Day view** — детальное представление одного дня с горизонтальной временной шкалой. Полезно для планирования сложного дня с множеством пересекающихся смен.

**Визуальная кодировка состояний** помогает быстро оценить расписание:
- Цвет фона определяется должностью (цветовая палитра для каждой Position)
- Прозрачность отличает черновики от опубликованных смен
- Серый цвет маркирует прошедшие смены
- Значок `2/3` показывает заполненность (назначено / capacity)

**Календарные представления сотрудника** используют ту же инфраструктуру, но с упрощённой логикой, ориентированной на read-only просмотр личных назначений.

Валидация формы выполняется на клиенте перед отправкой:

**Employee picker** — компонент выбора сотрудников для назначения на смену. Решаемая проблема: показывать только сотрудников, которые могут работать на данной смене. Реализация использует фильтрацию по должности:

```javascript
function filterEmployeePicker() {
  const positionId = getEl('shiftPosition')?.value;
  
  // Очищаем список
  for (const row of list.children) row.remove();
  
  if (!positionId) {
    list.appendChild(createEmptyMessage('Select position first'));
    return;
  }
  
  // Показываем только сотрудников выбранной должности
  const rows = employeeBuckets.get(positionId) || [];
  rows.forEach(row => list.appendChild(row));
  
  if (!rows.length) {
    list.appendChild(createEmptyMessage('No employees for this position'));
  }
}
```

Сотрудники группируются по должностям (buckets) при инициализации. При выборе должности смены отображаются только релевантные сотрудники. Это предотвращает ошибки и ускоряет работу — менеджеру не нужно искать сотрудников среди всех.

**Фильтры** позволяют настроить отображение смен: по должности, статусу (черновик/опубликовано), заполненности (недоукомплектованные). Фильтрация выполняется через GET-параметры URL:

```javascript
function buildManagerFiltersUrl() {
  const url = new URL(form.action, window.location.origin);
  const params = new URLSearchParams();
  
  // Собираем значения фильтров
  const data = new FormData(form);
  for (const [key, value] of data.entries()) {
    params.append(key, String(value));
  }
  
  // Убираем пустые значения
  if (!params.get('status')) params.delete('status');
  
  url.search = params.toString();
  return url.toString();
}
```

Фильтры реализованы без AJAX — изменение фильтра перезагружает страницу с новыми параметрами. Такой подход обеспечивает корректную работу навигации браузера (кнопки назад/вперёд) и возможность поделиться ссылкой с конкретными фильтрами.

**Функция Undo** позволяет отменить последнее массовое действие. Реализация использует сессию Django для хранения информации о последнем действии:

При вызове Undo система извлекает информацию из сессии и выполняет обратное действие.

```javascript
function computeEmployeePeriodStats(shifts) {
  const minutesByEmployeeId = new Map();
  shifts.forEach((s) => {
    const duration = parseTimeToMinutes(s.end_time) - parseTimeToMinutes(s.start_time);
    s.assigned_employee_ids.forEach((eid) => {

    });
  });
  return { minutesByEmployeeId };
}
```

```javascript
    updateCalendarCell(dateStr, data.is_unavailable);
  });
}
```

Менеджеры видят отмеченную сотрудником недоступность при назначении смен, что помогает избежать конфликтов расписания.

**Визуальная согласованность** между интерфейсами менеджера и сотрудника обеспечивается:
- Единым header-компонентом
- Общей цветовой палитрой позиций
- Идентичными календарными компонентами
- Согласованными стилями кнопок, карточек и модальных окон

Клиентская часть ShiftFlow демонстрирует, как чистый JavaScript и CSS могут обеспечить богатый интерактивный интерфейс без сложных фреймворков. Гибридная модель рендеринга сочетает преимущества серверного HTML и клиентской интерактивности, а модульная организация кода упрощает сопровождение.

## 4.7 Безопасность

Безопасность веб-приложения — критический аспект, требующий внимания на всех уровнях архитектуры. ShiftFlow работает с данными сотрудников и расписаниями, поэтому защита от несанкционированного доступа и типичных веб-уязвимостей является приоритетом. Этот раздел рассматривает механизмы аутентификации, авторизации и защиты от распространённых атак.

### 4.7.1 Аутентификация и авторизация

**Система аутентификации** построена на встроенных механизмах Django. Фреймворк предоставляет проверенную реализацию, включающую безопасное хранение паролей, управление сессиями и защиту от атак перебора.

Пароли хранятся в виде хешей с использованием алгоритма PBKDF2 с SHA256 и случайной солью. Django автоматически применяет этот алгоритм при вызове `user.set_password()`. Даже при компрометации базы данных злоумышленник не получит исходные пароли.

Валидация паролей обеспечивает минимальный уровень сложности:

```python
AUTH_PASSWORD_VALIDATORS = [
    {"NAME": "django.contrib.auth.password_validation.UserAttributeSimilarityValidator"},
    {"NAME": "django.contrib.auth.password_validation.MinimumLengthValidator"},
    {"NAME": "django.contrib.auth.password_validation.CommonPasswordValidator"},
    {"NAME": "django.contrib.auth.password_validation.NumericPasswordValidator"},
]
```

Эти валидаторы предотвращают использование паролей, похожих на имя пользователя, слишком коротких, распространённых или состоящих только из цифр.

**Генерация временных паролей** для новых сотрудников использует криптографически безопасный генератор:

```python
@staticmethod
def generate_temporary_password(length: int = 14) -> str:
    alphabet = string.ascii_letters + string.digits
    return "".join(secrets.choice(alphabet) for _ in range(length))
```

Модуль `secrets` предназначен для генерации криптографически стойких случайных значений, в отличие от `random`, который не подходит для целей безопасности. Длина 14 символов из алфавита 62 символов обеспечивает достаточную энтропию.

Временные учётные данные отображаются менеджеру только один раз, сразу после создания сотрудника. Они сохраняются в сессии и удаляются после отображения:

```python
def _store_one_time_credentials(request: HttpRequest, employee: User, temp_password: str) -> None:
    request.session["one_time_credentials"] = {
        "login": employee.email,
        "temporary_password": temp_password,
        "employee_id": employee.employee_id,
    }

# При отображении страницы:
creds = request.session.pop("one_time_credentials", None)
```

Метод `pop` извлекает и удаляет данные из сессии атомарно. При обновлении страницы учётные данные уже недоступны.

**Ролевая авторизация** ограничивает доступ к функциональности в зависимости от роли пользователя. Реализация через декораторы описана в разделе 4.5.3, здесь рассмотрим аспекты безопасности.

Декоратор `@manager_required` проверяет два условия:

```python
def role_required(user_attr: str, redirect_to: str):
    def decorator(view_func):
        @wraps(view_func)
        def _wrapped(request: HttpRequest, *args, **kwargs):
            if not request.user.is_authenticated:
                return redirect("login")
            if not getattr(request.user, user_attr, False):
                return redirect(redirect_to)
            return view_func(request, *args, **kwargs)
        return _wrapped
    return decorator
```

Использование `getattr` с значением по умолчанию `False` гарантирует безопасное поведение даже при неожиданных значениях атрибута. Если атрибут отсутствует или равен `None`, доступ будет отклонён.

**Проверка владельца ресурса** предотвращает доступ к чужим данным. Менеджер может работать только со сменами, которые он создал:

```python
shift = get_object_or_404(Shift.objects, pk=shift_id, created_by=request.user)
```

Функция `get_object_or_404` возвращает HTTP 404 как при отсутствии записи, так и при несовпадении владельца. Это важно с точки зрения безопасности: злоумышленник не может определить, существует ли ресурс с данным ID у другого пользователя — ответ одинаковый в обоих случаях.

Аналогичная проверка применяется при управлении сотрудниками:

```python
return get_object_or_404(User.objects, pk=user_id, role=UserRole.EMPLOYEE)
```

Менеджер может редактировать только пользователей с ролью EMPLOYEE, что предотвращает эскалацию привилегий — редактирование собственной записи или записей других менеджеров.

### 4.7.2 Защита от уязвимостей

Django предоставляет встроенную защиту от наиболее распространённых веб-уязвимостей. ShiftFlow использует эти механизмы и дополняет их на уровне приложения.

**Защита от CSRF (Cross-Site Request Forgery)** предотвращает выполнение действий от имени аутентифицированного пользователя без его ведома. Django реализует защиту через токены, уникальные для каждой сессии.

Middleware `CsrfViewMiddleware` включён в цепочку обработки:

```python
MIDDLEWARE = [
    # ...
    "django.middleware.csrf.CsrfViewMiddleware",
    # ...
]
```

Каждая форма, выполняющая POST-запрос, должна содержать CSRF-токен:

```html
<form method="post" action="{% url 'create_shift' %}">
    {% csrf_token %}
    <!-- поля формы -->
</form>
```

Тег `{% csrf_token %}` генерирует скрытое поле с токеном. При обработке POST-запроса Django проверяет соответствие токена значению в сессии. Запрос без валидного токена отклоняется с ошибкой 403.

В ShiftFlow все формы содержат CSRF-токен — это требование применяется к формам на страницах управления сменами, сотрудниками, а также на странице входа.

**Защита от XSS (Cross-Site Scripting)** предотвращает внедрение вредоносного JavaScript-кода через пользовательский ввод. Django автоматически экранирует переменные в шаблонах:

```html
<span class="employee-name">{{ employee.first_name }}</span>
```

Если `first_name` содержит `<script>alert('xss')</script>`, в HTML будет выведено `&lt;script&gt;alert('xss')&lt;/script&gt;` — браузер отобразит текст, а не выполнит скрипт.

В местах, где требуется явное экранирование (например, в атрибутах), используется фильтр `|escape`:

```html
<label data-employee-name="{{ e.get_full_name|default:e.username|escape }}">
```

На стороне JavaScript также применяется экранирование при динамическом формировании HTML:

```javascript
function escapeHtml(text) {
  const div = document.createElement('div');
  div.textContent = text;
  return div.innerHTML;
}

chip.innerHTML = `<span class="shift-chip-position-name">${escapeHtml(shift.position)}</span>`;
```

Функция `escapeHtml` использует встроенное поведение браузера: установка `textContent` и последующее чтение `innerHTML` автоматически экранирует специальные символы.

**Защита от SQL Injection** обеспечивается Django ORM. Все запросы к базе данных формируются через ORM, который использует параметризованные запросы:

```python
Shift.objects.filter(created_by_id=manager_id, date__gte=start)
```

Django преобразует это в SQL с параметрами:

```sql
SELECT * FROM shifts WHERE created_by_id = %s AND date >= %s
```

Значения передаются отдельно от запроса, что исключает возможность инъекции. Даже если `manager_id` содержит вредоносный SQL-код, он будет обработан как строковое значение, а не как часть запроса.

В ShiftFlow нет raw SQL-запросов — весь доступ к данным осуществляется через ORM.

**Защита от Clickjacking** предотвращает встраивание страницы приложения в iframe на вредоносном сайте. Middleware `XFrameOptionsMiddleware` добавляет заголовок:

```
X-Frame-Options: DENY
```

Браузер откажется отображать страницу внутри iframe, что предотвращает атаки, при которых пользователь кликает по невидимым элементам поверх легитимной страницы.

**SecurityMiddleware** добавляет дополнительные заголовки безопасности. В production-режиме рекомендуется включить:

- `SECURE_HSTS_SECONDS` — принудительное использование HTTPS
- `SECURE_CONTENT_TYPE_NOSNIFF` — предотвращение MIME-sniffing
- `SECURE_BROWSER_XSS_FILTER` — активация встроенного XSS-фильтра браузера

В текущей конфигурации эти настройки не активированы, так как приложение работает в режиме разработки. При развёртывании в production они должны быть включены.

**Валидация на стороне сервера** — последний рубеж защиты. Независимо от клиентской валидации, все данные проверяются на сервере:

```python
def clean_email(self):
    email = (self.cleaned_data.get("email") or "").strip().lower()
    if not email:
        raise ValidationError("Email is required.")
    # проверка уникальности
```

Клиентская валидация улучшает UX, но не является защитой — её легко обойти. Серверная валидация гарантирует целостность данных независимо от поведения клиента.

Описанные механизмы обеспечивают многоуровневую защиту ShiftFlow. Django предоставляет надёжный фундамент, а архитектурные решения приложения — проверка владельца ресурса, ролевая авторизация, экранирование — дополняют его на уровне бизнес-логики.

## 4.8 Демонстрация функциональности

Предыдущие разделы рассматривали архитектуру и реализацию отдельных компонентов ShiftFlow. Этот раздел демонстрирует, как компоненты работают вместе в контексте реальных сценариев использования. Описание сценариев позволяет увидеть систему глазами пользователя и проследить поток данных от интерфейса через бизнес-логику к базе данных.

### 4.8.1 Сценарий менеджера

Основной рабочий процесс менеджера включает три этапа: создание смен, назначение сотрудников и публикация расписания. Рассмотрим полный цикл планирования рабочей недели.

**Вход в систему и начальный экран.** Менеджер вводит email и пароль на странице входа. После успешной аутентификации Django создаёт сессию и перенаправляет на страницу `/manager/shifts/`. Декоратор `@manager_required` проверяет роль пользователя — сотрудник с ролью `employee` был бы перенаправлен на свой интерфейс.

Начальный экран отображает календарь в недельном представлении с текущей датой. Если смены ещё не созданы, календарная сетка пуста — только временные метки слева и дни недели сверху. Боковая панель показывает список сотрудников, сгруппированных по должностям.

**Создание смены.** Менеджер кликает на ячейку календаря, соответствующую понедельнику, 9:00. JavaScript перехватывает клик и открывает модальное окно создания смены. Форма предзаполнена: дата установлена в понедельник, время начала — 09:00. Менеджер выбирает должность «Barista» из выпадающего списка, устанавливает время окончания 14:00 и вместимость 2 (нужны два бариста на утреннюю смену).

При выборе должности компонент employee picker автоматически фильтрует список сотрудников. Отображаются только те, чья должность соответствует выбранной позиции смены. Менеджер отмечает двух сотрудников — Anna и Pavel.

При нажатии кнопки «Create» форма отправляется POST-запросом. На сервере происходит следующая цепочка:

1. Представление `create_shift` извлекает данные из `request.POST`
2. Создаётся объект `Shift` со статусом `DRAFT`
3. Вызывается сервисная функция `set_shift_assignments(shift, [anna_id, pavel_id])`
4. Сервис проверяет: соответствие позиции, вместимость, доступность сотрудников, отсутствие пересечений
5. Создаются записи `Assignment` для каждого сотрудника
6. Транзакция фиксируется, пользователь перенаправляется на календарь

На календаре появляется карточка смены. Она отображается полупрозрачной — визуальный индикатор статуса черновика. Значок «2/2» показывает полную укомплектованность.

**Планирование недели.** Менеджер продолжает создавать смены на всю неделю. Процесс повторяется: клик на ячейку → заполнение формы → создание. Для ускорения работы можно использовать кнопку «+ New Shift» в панели инструментов — она открывает модальное окно без предзаполненной даты.

При создании смены с пересечением система блокирует назначение. Например, если Anna уже назначена на смену 9:00–14:00, попытка назначить её на смену 12:00–17:00 в тот же день вызовет ошибку валидации. Сервер возвращает сообщение «Employee already assigned to overlapping shift», которое отображается как toast-уведомление.

**Редактирование смены.** Клик по карточке смены открывает модальное окно редактирования. JavaScript отправляет AJAX-запрос к endpoint `/manager/shifts/<id>/details/`, получает данные смены в формате JSON и заполняет форму. Менеджер может изменить время, переназначить сотрудников или удалить смену.

При сохранении изменений выполняется та же валидация, что и при создании. Если менеджер пытается уменьшить вместимость ниже количества назначенных сотрудников, система выводит ошибку «Cannot assign more employees than shift capacity».

**Публикация расписания.** После завершения планирования менеджер нажимает кнопку «Publish All» в панели инструментов. Система выполняет массовую операцию:

```python
published_ids, blocked_ids = publish_drafts_in_range(
    manager_id=request.user.id,
    start=week_start,
    end=week_end
)
```

Функция проверяет каждый черновик. Смены с назначенными сотрудниками, которые отметили недоступность на эту дату, блокируются от публикации. Остальные переводятся в статус `PUBLISHED`.

После публикации менеджер видит сообщение: «Published 12 shifts. 2 shifts blocked due to employee unavailability.» Заблокированные смены остаются черновиками — менеджер может переназначить сотрудников и опубликовать их отдельно.

Опубликованные смены отображаются непрозрачными — визуальное отличие от черновиков. Они становятся видны сотрудникам в их интерфейсе.

**Управление сотрудниками.** Параллельно с планированием смен менеджер может управлять сотрудниками через раздел `/manager/employees/`. При создании нового сотрудника система генерирует временный пароль и отображает учётные данные один раз. Менеджер должен передать их сотруднику — после закрытия модального окна пароль недоступен.

### 4.8.2 Сценарий сотрудника

Интерфейс сотрудника ориентирован на две задачи: просмотр назначенных смен и управление своей доступностью.

**Вход и просмотр смен.** Сотрудник входит в систему с учётными данными, полученными от менеджера. Декоратор `@employee_required` направляет его на страницу `/employee/shifts/`. Здесь отображается календарь с опубликованными сменами, на которые назначен данный сотрудник.

Календарь сотрудника проще, чем у менеджера: нет режима редактирования, нет боковой панели с другими сотрудниками. Карточки смен показывают время, должность и статус. Прошедшие смены отмечены серым цветом.

Представление формирует запрос с фильтрацией по текущему пользователю:

```python
shifts = Shift.objects.filter(
    status=ShiftStatus.PUBLISHED,
    assignments__employee=request.user
).select_related('position')
```

Метод `select_related` загружает связанную позицию в одном запросе, избегая N+1 проблемы при отображении списка смен.

**Управление недоступностью.** Раздел `/employee/unavailability/` позволяет сотруднику указать даты, когда он не может работать. Интерфейс представляет собой календарь с возможностью отмечать дни.

Клик по дате переключает её состояние: доступен ↔ недоступен. JavaScript отправляет POST-запрос, сервер создаёт или удаляет запись `EmployeeUnavailability`:

```python
if action == "add":
    EmployeeUnavailability.objects.get_or_create(employee=request.user, date=target_date)
elif action == "remove":
    EmployeeUnavailability.objects.filter(employee=request.user, date=target_date).delete()
```

Метод `get_or_create` предотвращает дублирование записей при повторных кликах.

**Влияние недоступности на планирование.** Когда сотрудник отмечает дату как недоступную, это влияет на работу менеджера двумя способами:

1. При назначении на смену в эту дату система выводит ошибку «Employee is unavailable»
2. При массовой публикации смены с этим сотрудником блокируются

Если сотрудник уже назначен на смену и затем отмечает эту дату как недоступную, смена не публикуется автоматически. Менеджер должен переназначить сотрудника или снять отметку недоступности.

### 4.8.3 Валидационные и граничные случаи

Система защищена от некорректного ввода на нескольких уровнях: клиентская валидация, серверная валидация форм и валидация бизнес-правил в сервисном слое.

**Валидация времени смены.** При создании смены система проверяет, что время окончания позже времени начала. Клиентская проверка срабатывает мгновенно:

```javascript
if (start >= end) {
    showToast('error', 'Invalid time range', 'End time must be after start time.');
    return;
}
```

Аналогичная проверка выполняется на сервере — клиентскую валидацию можно обойти, отправив запрос напрямую.

**Превышение вместимости.** При попытке назначить больше сотрудников, чем позволяет capacity, форма блокирует отправку. Если вместимость 2, а выбрано 3 сотрудника, появляется предупреждение. На сервере функция `validate_shift_capacity` генерирует `ValidationError`.

**Пересечение смен.** Назначение сотрудника на пересекающиеся смены блокируется. Алгоритм проверки учитывает частичное пересечение: смена 9:00–12:00 пересекается со сменой 11:00–15:00.

Сообщение об ошибке информативно: «Employee Anna already assigned to: Barista 11:00–15:00». Менеджер сразу понимает причину конфликта и может скорректировать назначение.

**Удаление связанных данных.** При попытке удалить должность, связанную со сменами, база данных отклоняет операцию благодаря `on_delete=models.PROTECT`. Система отображает сообщение о невозможности удаления. Решение — деактивировать должность через флаг `is_active`, а не удалять.

При удалении сотрудника его назначения удаляются каскадно (`on_delete=models.CASCADE`). Смены остаются, но без этого сотрудника — capacity освобождается для назначения других.

**Работа с прошедшими датами.** Система не запрещает создание смен на прошедшие даты — это может быть нужно для учёта фактически отработанного времени. Однако прошедшие смены визуально отличаются (серый цвет) и исключаются из массовых операций публикации.

**Дублирование назначений.** Уникальное ограничение на уровне базы данных (`unique_employee_per_shift`) предотвращает повторное назначение сотрудника на ту же смену. Даже если логика приложения пропустит дубликат, база данных его отклонит с ошибкой `IntegrityError`.

**Несуществующие ресурсы.** Запрос к несуществующей смене или смене другого менеджера возвращает HTTP 404. Функция `get_object_or_404` унифицирует обработку: пользователь не может определить, существует ли ресурс у другого менеджера.

**Неавторизованный доступ.** Попытка сотрудника открыть `/manager/shifts/` приводит к перенаправлению на `/employee/shifts/`. Декоратор проверяет роль и направляет в соответствующий интерфейс. Неаутентифицированный пользователь перенаправляется на страницу входа.

Демонстрация сценариев показывает, как архитектурные решения, описанные в предыдущих разделах, реализуются на практике. Разделение на слои обеспечивает надёжную валидацию: каждый уровень — клиент, представление, сервис, база данных — вносит свой вклад в целостность данных. Гибридная модель рендеринга обеспечивает быстрый отклик на действия пользователя при сохранении серверной верификации.

## 4.9 Оценка решения и возможности развития

Разработанное приложение ShiftFlow представляет собой функциональный прототип системы управления сменами. Оно решает поставленные задачи — планирование смен, назначение сотрудников, управление доступностью — однако, как и любой проект, имеет области для улучшения. Этот раздел критически оценивает текущее состояние системы и намечает направления дальнейшего развития.

### 4.9.1 Текущие ограничения

Честная оценка ограничений важна для понимания границ применимости системы и приоритизации будущей работы.

**Отсутствие автоматизированных тестов.** Наиболее существенное ограничение текущей реализации — отсутствие тестового покрытия. Все проверки выполнялись вручную в процессе разработки. Это создаёт несколько рисков:

- Регрессии при изменении кода — модификация одного компонента может незаметно сломать другой
- Сложность рефакторинга — без тестов трудно убедиться, что улучшения не нарушили существующую функциональность
- Отсутствие документированного поведения — тесты служат спецификацией ожидаемого поведения системы

Сервисный слой ShiftFlow особенно нуждается в тестировании. Функции валидации (`validate_employee_no_overlap`, `validate_shift_capacity`) содержат критичную бизнес-логику, ошибки в которой могут привести к некорректному планированию. Unit-тесты для этих функций имели бы высокую ценность при относительно низких затратах на написание.

**Отсутствие REST API.** Текущая архитектура ориентирована на серверный рендеринг с элементами AJAX. Полноценного REST API не существует — есть лишь несколько JSON-endpoints для получения данных смены. Это ограничивает возможности интеграции:

- Невозможно создать мобильное приложение, использующее тот же backend
- Затруднена интеграция с внешними системами (HR-системы, системы учёта рабочего времени)
- Сложнее реализовать SPA (Single Page Application), если такое решение будет принято в будущем

Текущий подход был осознанным выбором для прототипа — гибридный рендеринг проще в реализации и достаточен для базовой функциональности. Однако при масштабировании REST API становится необходимостью.

**Ограничения SQLite.** База данных SQLite выбрана для упрощения развёртывания — она не требует отдельного сервера и хранится в одном файле. Однако это создаёт ограничения для production-использования:

- Отсутствие параллельной записи — SQLite блокирует базу при записи, что может создать узкое место при одновременной работе нескольких менеджеров
- Ограниченная масштабируемость — при росте объёма данных производительность может деградировать
- Отсутствие расширенных типов данных — например, нативной поддержки JSON-полей для хранения дополнительных метаданных

Для текущего масштаба (один-два менеджера, десятки сотрудников) SQLite достаточен. При переходе к production-развёртыванию с множеством пользователей потребуется миграция на PostgreSQL или аналогичную СУБД.

**Отсутствие системы уведомлений.** Сотрудники не получают уведомлений о новых назначениях или изменениях в расписании. Они должны самостоятельно проверять интерфейс. В реальном использовании это создаёт риск пропуска смены из-за незнания об изменениях.

Реализация уведомлений требует выбора канала доставки (email, push-уведомления, SMS), настройки асинхронной обработки (Celery или аналог) и инфраструктуры для отправки. Эти компоненты выходили за рамки прототипа.

**Ограниченная ролевая модель.** Текущая система поддерживает только две роли: менеджер и сотрудник. В реальных организациях может потребоваться более гранулярное разделение:

- Старший менеджер с правом управления другими менеджерами
- Сотрудник с правом самостоятельного выбора смен из доступных
- Администратор системы для технических настроек

Расширение ролевой модели потребует модификации модели `User`, системы декораторов и интерфейсов.

**Отсутствие аналитики и отчётности.** Система не предоставляет инструментов для анализа данных: статистики по сменам, отчётов об отработанном времени, аналитики загруженности. Менеджер видит только текущее состояние расписания без возможности агрегированного анализа.

### 4.9.2 Рекомендации по дальнейшему развитию

На основе выявленных ограничений можно сформулировать приоритетные направления развития системы.

**Внедрение автоматизированного тестирования.** Первоочередная задача — создание тестового покрытия для критичных компонентов. Рекомендуемый подход:

1. Unit-тесты для сервисного слоя — функции валидации и бизнес-операции
2. Интеграционные тесты для представлений — проверка корректности HTTP-ответов и редиректов
3. End-to-end тесты для ключевых сценариев — создание смены, публикация, назначение

Django предоставляет встроенный тестовый фреймворк на базе Python unittest. Для сервисного слоя тесты могли бы выглядеть так:

```python
class TestShiftValidation(TestCase):
    def test_overlap_detection(self):
        # Создание тестовых данных
        shift1 = Shift.objects.create(date=date.today(), start_time=time(9, 0), end_time=time(12, 0), ...)
        shift2 = Shift.objects.create(date=date.today(), start_time=time(11, 0), end_time=time(15, 0), ...)
        
        # Назначение на первую смену
        Assignment.objects.create(shift=shift1, employee=self.employee)
        
        # Попытка назначения на пересекающуюся смену должна вызвать ошибку
        with self.assertRaises(ValidationError):
            validate_employee_no_overlap(self.employee.id, shift2)
```

Инвестиция в тесты окупается при первом же рефакторинге — уверенность в корректности изменений значительно ускоряет разработку.

**Разработка REST API.** Добавление полноценного API расширит возможности интеграции. Django REST Framework (DRF) — стандартный выбор для Django-проектов. Он предоставляет:

- Сериализаторы для преобразования моделей в JSON
- ViewSet-классы для стандартных CRUD-операций
- Встроенную аутентификацию (Token, JWT, Session)
- Автоматическую генерацию документации (OpenAPI/Swagger)

Архитектура ShiftFlow уже подготовлена к добавлению API благодаря сервисному слою. Бизнес-логика инкапсулирована в `services.py` и может использоваться как традиционными представлениями, так и API-endpoints:

```python
# Текущее представление
@manager_required
def create_shift(request):
    # ...извлечение данных...
    set_shift_assignments(shift, employee_ids)  # сервисный слой
    return redirect(...)

# Будущий API endpoint
class ShiftViewSet(viewsets.ModelViewSet):
    def perform_create(self, serializer):
        shift = serializer.save()
        employee_ids = self.request.data.get('employee_ids', [])
        set_shift_assignments(shift, employee_ids)  # тот же сервисный слой
```

Такой подход позволяет избежать дублирования бизнес-логики между веб-интерфейсом и API.

**Миграция на PostgreSQL.** При переходе к production-развёртыванию рекомендуется заменить SQLite на PostgreSQL. Преимущества:

- Поддержка параллельных соединений без блокировок
- Расширенные типы данных (JSONField, ArrayField)
- Полнотекстовый поиск для поиска сотрудников
- Надёжность и масштабируемость для enterprise-использования

Django ORM абстрагирует различия между СУБД, поэтому миграция не требует изменения кода приложения — достаточно изменить настройки в `settings.py`:

```python
DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.postgresql',
        'NAME': os.environ.get('DB_NAME'),
        'USER': os.environ.get('DB_USER'),
        'PASSWORD': os.environ.get('DB_PASSWORD'),
        'HOST': os.environ.get('DB_HOST', 'localhost'),
        'PORT': os.environ.get('DB_PORT', '5432'),
    }
}
```

Единственное требование — PostgreSQL-сервер и библиотека `psycopg2` для Python.

**Реализация PWA (Progressive Web App).** Преобразование ShiftFlow в PWA улучшит пользовательский опыт на мобильных устройствах:

- Установка на домашний экран без магазина приложений
- Работа в offline-режиме (просмотр расписания без сети)
- Push-уведомления о новых назначениях

Технически PWA требует:
- Service Worker для кеширования и offline-поддержки
- Web App Manifest для метаданных приложения
- HTTPS для безопасной работы Service Worker

Гибридная архитектура ShiftFlow хорошо подходит для PWA — серверный рендеринг обеспечивает быструю первую загрузку, а Service Worker может кешировать статические ресурсы и данные расписания.

**Система уведомлений.** Для production-использования необходимо уведомлять сотрудников об изменениях. Рекомендуемая архитектура:

1. Celery для асинхронной обработки задач
2. Redis или RabbitMQ как брокер сообщений
3. Email-уведомления через SMTP или сервис (SendGrid, Mailgun)
4. Опционально: push-уведомления через Firebase Cloud Messaging

При публикации смены система помещает задачу уведомления в очередь:

```python
@transaction.atomic
def publish_drafts_in_range(...):
    # ...существующая логика публикации...
    
    # Асинхронная отправка уведомлений
    for shift_id in published_ids:
        send_shift_notification.delay(shift_id)  # Celery task
```

Асинхронная обработка не блокирует ответ пользователю — менеджер сразу видит результат публикации, а уведомления отправляются в фоновом режиме.

**Расширенная аналитика.** Добавление раздела аналитики повысит ценность системы для менеджеров:

- Статистика по сотрудникам: количество смен, отработанные часы
- Анализ загруженности: дни и часы с недостаточным/избыточным покрытием
- История изменений: кто и когда модифицировал расписание

Реализация может использовать встроенные возможности Django ORM для агрегации:

```python
from django.db.models import Count, Sum
from django.db.models.functions import TruncWeek

# Статистика смен по неделям
Shift.objects.filter(status='published').annotate(
    week=TruncWeek('date')
).values('week').annotate(
    shift_count=Count('id'),
    total_capacity=Sum('capacity')
)
```

Приведённые рекомендации представляют собой эволюционный путь развития системы. Каждое улучшение может быть реализовано независимо, без полной переработки существующего кода. Модульная архитектура ShiftFlow, разделение на слои и использование стандартных паттернов Django создают прочную основу для дальнейшего роста функциональности.
